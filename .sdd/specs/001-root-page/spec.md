# Feature Spec: 루트 페이지 (`/`)

> **Feature ID**: 001
> **Status**: Draft
> **Created**: 2026-03-02
> **Reference**: `reference.html`

---

## 1. 기능 개요

루트 경로(`/`)에 접속한 사용자의 인증 상태에 따라 완전히 다른 페이지를 제공한다.

- **비로그인 사용자**: 서비스 소개와 로그인/회원가입 유도가 포함된 랜딩 페이지
- **로그인 사용자**: AI 스킬을 검색하고 추천 스킬을 탐색할 수 있는 대시보드

---

## 2. 유저 스토리

### US-001: 비로그인 사용자의 첫 방문
> As a 비로그인 사용자,
> I want to 서비스가 무엇인지 한눈에 파악하고 가입/로그인할 수 있기를,
> So that 서비스를 이해하고 빠르게 시작할 수 있다.

### US-002: 로그인 사용자의 대시보드 접근
> As a 로그인 사용자,
> I want to 로그인 후 곧바로 스킬 탐색과 검색이 가능한 대시보드를 보기를,
> So that 필요한 스킬을 빠르게 찾아 업무에 활용할 수 있다.

### US-003: 사이드바를 통한 네비게이션
> As a 로그인 사용자,
> I want to 사이드바에서 메인 메뉴와 카테고리를 통해 원하는 영역으로 이동하기를,
> So that 플랫폼 내에서 원하는 기능에 빠르게 접근할 수 있다.

### US-004: 스킬 검색
> As a 로그인 사용자,
> I want to 검색바에 키워드를 입력하여 스킬을 검색하기를,
> So that 이름이나 키워드로 원하는 스킬을 빠르게 찾을 수 있다.

## 3. 기능 요구사항

### 3.1 인증 기반 라우팅

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-01 | 비로그인 사용자가 `/`에 접속하면 랜딩 페이지를 표시한다 | 필수 |
| FR-02 | 로그인 사용자가 `/`에 접속하면 대시보드를 표시한다 | 필수 |
| FR-03 | 인증 상태 확인 중에는 로딩 상태를 표시한다 | 필수 |

### 3.2 랜딩 페이지 (비로그인)

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-04 | 서비스명("AI 스킬 허브")과 한 줄 소개 문구를 표시한다 | 필수 |
| FR-05 | "로그인" 버튼을 제공하여 로그인 페이지로 이동할 수 있다 | 필수 |
| FR-06 | "회원가입" 버튼을 제공하여 회원가입 페이지로 이동할 수 있다 | 필수 |
| FR-07 | 서비스의 핵심 가치를 간략히 설명하는 문구를 포함한다 | 권장 |

### 3.3 대시보드 레이아웃 (로그인)

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-08 | 좌측에 고정 사이드바를 표시한다 | 필수 |
| FR-09 | 상단에 헤더 영역을 표시한다 | 필수 |
| FR-10 | 사이드바와 헤더를 제외한 나머지 영역에 메인 콘텐츠를 표시한다 | 필수 |

### 3.4 사이드바

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-11 | 서비스 로고와 서비스명을 표시한다 | 필수 |
| FR-12 | "메인" 그룹에 대시보드, 마켓플레이스, 내 에이전트 메뉴를 표시한다 | 필수 |
| FR-13 | "카테고리" 그룹에 업무 자동화, 문서 작성, 데이터 분석, 코딩 지원 메뉴를 표시한다 | 필수 |
| FR-14 | 현재 선택된 메뉴를 시각적으로 구분하여 표시한다 (활성 상태) | 필수 |
| FR-15 | 하단에 "에이전트 생성" 버튼을 표시한다 | 필수 |
| FR-16 | 하단에 "도움말" 링크를 표시한다 | 권장 |

## 5. 가정 사항

1. Supabase Auth를 통한 인증 시스템이 이미 구축되어 있거나 별도 명세로 구현될 예정이다
2. 추천 스킬 데이터는 데이터베이스에서 조회하며, 초기에는 하드코딩된 목업 데이터를 사용할 수 있다
3. 검색 기능은 이 명세에서 UI만 정의하며, 실제 검색 로직은 별도 명세로 다룬다
4. 사이드바의 각 메뉴 클릭 시 이동할 상세 페이지들은 별도 명세로 구현된다
5. 알림 기능의 상세 동작은 별도 명세로 다룬다

## 6. 제약 조건

1. Constitution에 정의된 DDD 아키텍처 원칙(도메인 → 애플리케이션 → 인프라 계층 분리)을 준수해야 한다
2. `any` 타입 사용을 금지한다
3. TypeScript strict 모드를 준수해야 한다
4. 외부 UI 라이브러리 도입 없이 Tailwind CSS로 구현한다 (reference.html 기준)
5. Next.js App Router를 사용한다

## 7. 수용 기준 체크리스트

### 랜딩 페이지 (비로그인)
- [ ] 비로그인 상태에서 `/` 접속 시 랜딩 페이지가 표시된다
- [ ] 서비스명과 소개 문구가 표시된다
- [ ] 로그인/회원가입 버튼이 표시되고 클릭 시 올바른 경로로 이동한다

### 대시보드 (로그인)
- [ ] 로그인 상태에서 `/` 접속 시 대시보드가 표시된다
- [ ] 사이드바에 모든 메뉴 항목이 표시되고, 현재 페이지가 활성 상태로 표시된다
- [ ] 헤더에 브레드크럼, 사용자 프로필이 표시된다
- [ ] 스킬 검색 입력 필드가 표시되고 입력이 가능하다
- [ ] 추천 스킬이 카드 그리드 형태로 표시된다
- [ ] 스킬 카드에 아이콘, 이름, 설명, 카테고리 태그가 모두 표시된다

### 공통
- [ ] 다크 모드가 정상 동작한다
- [ ] 반응형 레이아웃이 모바일~데스크탑에서 정상 표시된다
- [ ] 인증 상태 확인 중 로딩 상태가 표시된다 (화면 깜빡임 없음)
- [ ] 페이지 로드 시간이 3초 이내이다
