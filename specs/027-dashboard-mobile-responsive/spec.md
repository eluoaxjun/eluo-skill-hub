## User Scenarios & Testing *(mandatory)*

### User Story 1 - 모바일에서 대시보드 탐색 (Priority: P1)

모바일 사용자가 대시보드에 접속하면 사이드바가 화면을 가리지 않고, 햄버거 메뉴 버튼을 통해 사이드바를 열고 닫을 수 있어야 한다. 사이드바가 열리면 오버레이 형태로 표시되며, 메뉴 항목을 선택하면 자동으로 닫힌다.

**Why this priority**: 사이드바가 모바일에서 항상 고정(w-72)으로 표시되어 콘텐츠 영역을 거의 사용할 수 없는 상태이므로, 가장 먼저 해결해야 할 핵심 문제이다.

**Independent Test**: 모바일 뷰포트(375px)에서 페이지 로드 시 사이드바가 숨겨져 있고, 햄버거 버튼으로 열고 닫을 수 있는지 확인하여 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 모바일 뷰포트(< 768px)에서 대시보드 접속, **When** 페이지가 로드되면, **Then** 사이드바가 숨겨져 있고 햄버거 메뉴 버튼이 헤더에 표시된다
2. **Given** 사이드바가 닫힌 상태, **When** 햄버거 버튼을 탭하면, **Then** 사이드바가 오버레이로 슬라이드인되고 배경이 어두워진다
3. **Given** 사이드바가 열린 상태, **When** 카테고리 메뉴를 탭하면, **Then** 해당 카테고리로 필터링되고 사이드바가 자동으로 닫힌다
4. **Given** 사이드바가 열린 상태, **When** 오버레이 배경을 탭하면, **Then** 사이드바가 닫힌다

---

### User Story 2 - 모바일 검색바 및 콘텐츠 영역 최적화 (Priority: P1)

모바일 사용자가 검색바와 스킬 카드를 불편함 없이 사용할 수 있도록 패딩, 폰트 크기, 버튼 크기 등이 모바일 화면에 맞게 조정되어야 한다.

**Why this priority**: 검색바와 스킬 카드는 대시보드의 핵심 기능이며, 현재 데스크톱 전용 크기(p-10, text-4xl, pr-36)가 모바일에서 레이아웃 깨짐을 유발한다.

**Independent Test**: 모바일 뷰포트에서 검색바에 텍스트를 입력하고 검색 버튼을 탭하여 스킬 카드가 올바르게 표시되는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 모바일 뷰포트, **When** 검색바 영역이 렌더링되면, **Then** 제목 텍스트, 입력 필드, 검색 버튼이 화면 너비 내에 잘려짐 없이 표시된다
2. **Given** 모바일 뷰포트, **When** 스킬 카드 그리드가 렌더링되면, **Then** 카드가 1열로 표시되며 적절한 패딩과 간격을 가진다
3. **Given** 모바일 뷰포트, **When** 스킬 카드를 탭하면, **Then** 스킬 상세 페이지로 정상 이동한다

### User Story 3 - 모바일 헤더 최적화 (Priority: P2)

모바일에서 헤더가 콤팩트하게 표시되어 화면 공간을 효율적으로 사용할 수 있어야 한다.

**Why this priority**: 헤더가 과도한 패딩(px-10, h-20)을 가지고 있어 모바일에서 콘텐츠 영역을 줄이지만, 기능적 문제보다는 공간 효율성 개선이므로 P2로 지정한다.

**Independent Test**: 모바일 뷰포트에서 헤더의 높이와 내부 요소 배치가 적절한지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 모바일 뷰포트, **When** 헤더가 렌더링되면, **Then** 높이가 줄어들고 패딩이 모바일에 적합하게 조정된다
2. **Given** 모바일 뷰포트, **When** 헤더가 렌더링되면, **Then** 햄버거 메뉴 버튼, 브레드크럼, 프로필 드롭다운이 겹치지 않고 표시된다

---

### Edge Cases

- 사이드바 열림 상태에서 화면을 회전(가로<->세로)하면 어떻게 되는가? -> 뷰포트 변경 시 사이드바 상태를 적절히 처리한다
- 768px 근처의 뷰포트에서 데스크톱/모바일 전환이 매끄러운가? -> 브레이크포인트 전환 시 깜빡임이나 레이아웃 깨짐이 없어야 한다
- 매우 긴 카테고리 이름이 모바일 사이드바에서 잘리는 경우는? -> 텍스트가 말줄임(ellipsis) 처리된다
- 스킬 카드의 긴 설명이나 많은 태그가 모바일에서 레이아웃을 깨뜨리는 경우는? -> 기존 line-clamp와 태그 제한(3개)이 유지된다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 768px 미만 뷰포트에서 사이드바를 기본적으로 숨기고, 햄버거 메뉴 버튼을 통해 토글할 수 있어야 한다
- **FR-002**: 모바일 사이드바는 오버레이 형태로 표시되며, 배경 클릭 시 닫혀야 한다
- **FR-003**: 모바일 사이드바에서 메뉴 항목 선택 시 사이드바가 자동으로 닫혀야 한다
- **FR-004**: 검색바의 제목, 입력 필드, 버튼이 모바일 뷰포트에서 화면 너비를 초과하지 않아야 한다
- **FR-005**: 스킬 카드 그리드가 모바일에서 1열 레이아웃으로 표시되어야 한다
- **FR-006**: 헤더가 모바일에서 콤팩트한 높이와 패딩을 가져야 한다
- **FR-007**: 모든 터치 타겟(버튼, 링크)이 최소 44x44px 이상의 터치 영역을 가져야 한다
- **FR-008**: 데스크톱(768px 이상)에서는 기존 레이아웃이 변경 없이 유지되어야 한다
- **FR-009**: 모바일/데스크톱 전환 시 레이아웃 깨짐이나 깜빡임이 없어야 한다

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 모바일 뷰포트(375px)에서 모든 UI 요소가 화면 너비를 초과하지 않고 가로 스크롤이 발생하지 않는다
- **SC-002**: 모바일에서 사이드바 토글 애니메이션이 300ms 이내에 완료된다
- **SC-003**: 데스크톱 뷰포트(1280px)에서 기존 레이아웃과 동일하게 렌더링된다
- **SC-004**: 모바일에서 검색, 카테고리 필터링, 스킬 카드 탭 등 핵심 기능이 100% 동작한다
- **SC-005**: Playwright 자동화 테스트가 모바일(375px) 및 데스크톱(1280px) 뷰포트에서 모두 통과한다

## Assumptions

- 모바일 브레이크포인트는 Tailwind CSS의 기본값인 768px(md)을 사용한다
- 주요 테스트 대상 모바일 뷰포트는 375px(iPhone SE/12/13/14)이다
- 태블릿(768px~1024px)은 데스크톱 레이아웃을 따른다
- 기존 데스크톱 디자인과 기능은 일체 변경하지 않는다 (모바일 스타일만 추가)
- 사이드바 오버레이의 애니메이션은 CSS transition 또는 Tailwind 유틸리티로 구현한다
