# Feature Specification: 데이터 캐싱 전략 수립 및 최적화

**Feature Branch**: `019-data-caching-strategy`
**Created**: 2026-03-04
**Status**: Draft
**Input**: User description: "현재 프로젝트가 서버사이드렌더링을 기반으로 동작하는데, 데이터 캐싱처리가 제대로 안된것같아. 분석 후 클라이언트사이드렌더링을 적용할 수 있는 공간이 있다면 어떤 라이브러리를 쓰면되는지, 어떻게 변경할건지에 대해서 알려줘."

## 현재 상태 분석 결과

### 발견된 문제점

1. **대시보드 페이지 순차 쿼리**: 스킬 목록 → 북마크 목록 → 사용자 역할 조회가 순차적으로 실행되어 페이지 로딩 시간이 누적됨
2. **캐싱 설정 부재**: 서버 컴포넌트에 명시적인 캐싱 전략이 없어 매 요청마다 전체 데이터를 다시 조회함
3. **피드백 제출 후 무효화 누락**: 피드백/답글 작성 서버 액션에서 경로 재검증이 호출되지 않아 데이터 정합성 문제 발생 가능
4. **클라이언트 캐싱 한계**: 스킬 상세 팝업의 인메모리 캐시가 페이지 새로고침 시 소실됨

### 분석 결론

현재 프로젝트는 서버사이드 렌더링 기반으로 동작하며, 사용자별 인증 데이터를 포함하므로 **완전한 정적 캐싱(ISR)은 부적합**하다. 그러나 다음 두 영역에서 의미있는 개선이 가능하다:

1. **서버 측**: 순차 쿼리를 병렬화하고 데이터 무효화를 올바르게 적용
2. **클라이언트 측**: 별도의 CSR 라이브러리(React Query, SWR 등)를 도입할 필요 없이, 기존 서버 액션 + 서버 컴포넌트 패턴을 최적화하는 것이 현 규모에 적합

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 대시보드 페이지 빠른 로딩 (Priority: P1)

사용자가 대시보드 페이지에 접속하면, 스킬 목록·북마크·사용자 역할 정보가 지금보다 빠르게 표시된다. 현재 세 개의 데이터 조회가 하나씩 차례로 실행되는데, 독립적인 조회를 동시에 실행하여 전체 대기 시간을 줄인다.

**Why this priority**: 대시보드는 사용자가 가장 자주 방문하는 페이지이며, 로딩 속도 개선이 전체 서비스 체감 성능에 가장 큰 영향을 미침

**Independent Test**: 대시보드 페이지 접속 시 스킬 목록, 북마크 표시, 역할 기반 UI가 모두 정상 표시되는지 확인. 로딩 시간이 이전 대비 단축되었는지 측정.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 대시보드에 접속할 때, **When** 페이지가 로드되면, **Then** 스킬 목록, 북마크 상태, 사용자 역할이 모두 정상적으로 표시된다
2. **Given** 로그인한 사용자가 대시보드에 접속할 때, **When** 페이지 로딩 시간을 측정하면, **Then** 이전 대비 30% 이상 빠르게 표시된다
3. **Given** 비로그인 사용자가 대시보드에 접속할 때, **When** 페이지가 로드되면, **Then** 스킬 목록이 정상 표시되고 북마크/역할 관련 기능은 비활성화된다

---

### User Story 2 - 데이터 변경 후 화면 즉시 반영 (Priority: P2)

사용자가 피드백 작성, 답글 작성 등 데이터를 변경하는 행위를 수행한 후, 관련 페이지에 돌아왔을 때 변경사항이 정확하게 반영되어 있다. 현재 일부 서버 액션에서 데이터 무효화가 누락되어 최신 데이터가 표시되지 않는 문제를 해결한다.

**Why this priority**: 데이터 정합성은 사용자 신뢰에 직결되며, 작성한 피드백이 즉시 보이지 않으면 혼란을 야기함

**Independent Test**: 피드백을 작성한 뒤 대시보드로 돌아가서 해당 스킬 팝업을 다시 열었을 때 새 피드백이 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 스킬 상세 팝업에서 피드백을 작성할 때, **When** 피드백이 제출되면, **Then** 해당 피드백이 목록에 즉시 나타난다
2. **Given** 사용자가 피드백에 답글을 작성할 때, **When** 답글이 제출되면, **Then** 해당 답글이 피드백 하위에 즉시 나타난다
3. **Given** 사용자가 북마크를 토글할 때, **When** 내 에이전트 페이지로 이동하면, **Then** 변경된 북마크 상태가 정확하게 반영되어 있다

---

### User Story 3 - 사용자 역할 정보 효율적 조회 (Priority: P3)

대시보드 페이지에서 매번 사용자 역할(viewer/admin 등)을 조회하는 별도 쿼리를 줄이고, 레이아웃 수준에서 한 번 조회한 역할 정보를 하위 페이지에서 재사용한다. 이를 통해 페이지 이동 시마다 반복되는 불필요한 조회를 제거한다.

**Why this priority**: 역할 정보는 세션 중 거의 변하지 않으므로 캐싱 효과가 높고, 모든 포털 페이지에서 반복 조회를 제거할 수 있음

**Independent Test**: 대시보드 → 내 에이전트 → 대시보드 페이지를 이동하며 역할 기반 UI(viewer 제한 등)가 일관되게 동작하는지 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 포털 영역 내 페이지를 이동할 때, **When** 각 페이지가 로드되면, **Then** 사용자 역할에 따른 UI 제한이 일관되게 적용된다
2. **Given** 사용자가 대시보드에서 내 에이전트 페이지로 이동할 때, **When** 네트워크 요청을 관찰하면, **Then** 역할 조회를 위한 추가 쿼리가 발생하지 않는다

---

### Edge Cases

- 사용자 인증 세션이 만료된 상태에서 대시보드 접속 시 어떻게 되는가? → 로그인 페이지로 리다이렉트
- 병렬 쿼리 중 하나가 실패하면 어떻게 되는가? → 실패한 영역만 기본값(빈 배열 등)으로 처리하고, 나머지 데이터는 정상 표시
- 네트워크가 느린 환경에서 병렬 쿼리가 동시에 느려지면? → 가장 느린 쿼리가 전체 지연을 결정하지만, 순차 대비 여전히 빠름
- 서버 액션 실행 중 사용자가 페이지를 이탈하면? → 서버 측에서 작업은 완료되며 다음 방문 시 반영됨

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 대시보드 페이지의 스킬 목록, 북마크 목록, 사용자 역할 조회가 동시에 실행되어야 한다
- **FR-002**: 피드백 제출 서버 액션이 실행된 후, 관련 페이지의 캐시가 무효화되어야 한다
- **FR-003**: 답글 제출 서버 액션이 실행된 후, 관련 페이지의 캐시가 무효화되어야 한다
- **FR-004**: 병렬 쿼리 중 개별 쿼리가 실패해도 나머지 데이터는 정상적으로 표시되어야 한다
- **FR-005**: 사용자 역할 정보는 포털 레이아웃 수준에서 한 번 조회되어 하위 페이지에서 재사용 가능해야 한다
- **FR-006**: 기존의 검색, 카테고리 필터, 페이지네이션 기능이 최적화 후에도 동일하게 동작해야 한다
- **FR-007**: 스킬 상세 팝업의 클라이언트 캐시는 기존 동작(5분 TTL, SWR 패턴)을 유지해야 한다

### Key Entities

- **대시보드 스킬 목록**: 공개된 스킬의 요약 정보 (제목, 설명, 카테고리, 평점, 작성자)
- **북마크 목록**: 사용자가 즐겨찾기한 스킬 ID 목록
- **사용자 역할**: 사용자의 권한 수준 (viewer, admin 등)
- **피드백/답글**: 스킬에 대한 사용자 리뷰와 댓글

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 대시보드 페이지 로딩 시간이 최적화 전 대비 30% 이상 단축된다
- **SC-002**: 피드백/답글 작성 후 100%의 경우 관련 페이지에서 최신 데이터가 표시된다
- **SC-003**: 포털 내 페이지 이동 시 사용자 역할 조회 횟수가 1회로 감소한다 (현재 페이지당 1회 → 레이아웃당 1회)
- **SC-004**: 모든 기존 기능(검색, 필터, 북마크, 피드백, 다운로드)이 최적화 전과 동일하게 동작한다

## Assumptions

- 현 프로젝트 규모(사용자 수, 데이터 양)에서 React Query/SWR 등 별도 클라이언트 캐싱 라이브러리 도입은 오버엔지니어링으로 판단하며, 서버 컴포넌트 + 서버 액션 패턴의 최적화로 충분하다
- 사용자 역할은 세션 중 변경되지 않는다고 가정한다
- 스킬 상세 팝업의 인메모리 SWR 캐시는 018-skill-popup-optimization에서 이미 구현되었으므로, 이번 피처에서는 유지만 한다
- 서버 측 ISR(Incremental Static Regeneration)은 사용자별 인증 데이터가 포함되어 있어 적용하지 않는다

## Scope Boundaries

### In Scope

- 대시보드 서버 컴포넌트의 쿼리 병렬화
- 서버 액션의 데이터 무효화(revalidation) 누락 수정
- 사용자 역할 조회 최적화 (레이아웃 수준 공유)

### Out of Scope

- React Query, SWR 등 클라이언트 캐싱 라이브러리 도입 (현 규모에 불필요)
- 서버 측 Redis 캐싱 레이어 추가 (인프라 변경 필요)
- CDN 캐싱 설정 (Vercel 배포 설정 변경 필요)
- DB 인덱스 최적화 (DBA 영역)
- 스킬 상세 팝업 캐싱 (018-skill-popup-optimization에서 완료)
