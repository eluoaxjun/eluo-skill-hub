# Implementation Plan

- [ ] 1. Supabase SSR 클라이언트 인프라 구성
- [ ] 1.1 `@supabase/ssr` 패키지를 설치하고, 기존 브라우저 클라이언트를 `createBrowserClient` 기반으로 리팩터링한다
  - `@supabase/ssr` 의존성을 추가한다
  - 기존 `createClient` 직접 호출 방식을 `@supabase/ssr`의 `createBrowserClient`를 래핑하는 팩토리 함수로 전환한다
  - 기존 코드에서 Supabase 싱글톤을 참조하는 부분이 있다면 새 팩토리 함수 호출로 교체한다
  - _Requirements: 4.2_

- [ ] 1.2 서버 컴포넌트와 서버 액션에서 사용할 Supabase 서버 클라이언트 팩토리 함수를 생성한다
  - `@supabase/ssr`의 `createServerClient`를 래핑하여 Next.js `cookies()` API 기반으로 쿠키를 읽고 쓰는 서버 클라이언트를 제공한다
  - 매 호출마다 새 인스턴스를 생성하여 요청 간 세션 격리를 보장한다
  - 환경변수 `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_DEFAULT_KEY`를 사용한다
  - _Requirements: 4.2_

- [ ] 2. user-account 도메인 계층 구현
- [ ] 2.1 (P) eluocnc.com 도메인 제한 이메일 값 객체를 구현한다
  - 이메일 형식 유효성을 검증하고 `@eluocnc.com` 도메인만 허용하는 불변 값 객체를 생성한다
  - 유효하지 않은 이메일이나 허용되지 않은 도메인이 전달되면 생성 시점에 예외를 발생시킨다
  - 정적 검증 메서드를 제공하여 생성 전 유효성 확인이 가능하도록 한다
  - 기존 shared 모듈의 ValueObject 기반 클래스를 상속한다
  - _Requirements: 2.3, 2.4_

- [ ] 2.2 (P) 패스워드 정책 값 객체를 구현한다
  - 최소 8자 이상, 특수문자 1개 이상 포함을 강제하는 불변 값 객체를 생성한다
  - 정책 위반 시 생성 시점에 예외를 발생시킨다
  - 정적 검증 메서드를 제공한다
  - 기존 shared 모듈의 ValueObject 기반 클래스를 상속한다
  - _Requirements: 2.5_

- [ ] 2.3 (P) 인증 결과 타입과 인증 리포지토리 인터페이스를 정의한다
  - 인증 성공, 에러, 가입 성공 상태를 타입 안전하게 표현하는 판별 합집합 타입을 정의한다
  - 에러 코드 타입(잘못된 자격증명, 이메일 미인증, 이메일 중복 등록, 도메인 위반, 네트워크 오류, 알 수 없는 오류)을 정의한다
  - 로그인, 회원가입, 로그아웃, OTP 검증 메서드를 갖는 인증 리포지토리 인터페이스를 정의한다
  - 도메인 계층이므로 외부 라이브러리 의존 없이 순수 TypeScript로 작성한다
  - _Requirements: 1.3, 1.7, 1.8, 1.9, 2.7, 2.9, 4.2, 4.3, 5.3_

- [ ] 3. user-account 애플리케이션 계층 구현
- [ ] 3.1 로그인 유스케이스를 구현한다
  - 이메일과 패스워드를 받아 인증 리포지토리를 통해 로그인을 수행하고 인증 결과를 반환한다
  - 성공 시 메인 페이지 리다이렉트 경로를 포함한 결과를 반환한다
  - 에러 코드에 따라 적절한 사용자 메시지를 매핑한다(잘못된 자격증명, 이메일 미인증 등)
  - 2단계에서 정의한 인증 리포지토리 인터페이스에 의존한다
  - _Requirements: 1.3, 1.4, 1.7, 1.8, 1.9_

- [ ] 3.2 회원가입 유스케이스를 구현한다
  - 이메일과 패스워드를 받아 서버 사이드에서 도메인 이메일 값 객체와 패스워드 값 객체로 검증한 후 회원가입을 수행한다
  - 도메인 검증 실패 시 적절한 에러 결과를 반환한다
  - 중복 이메일 감지 시 이미 등록된 이메일임을 안내하는 결과를 반환한다
  - 성공 시 인증 메일 발송 안내 메시지를 포함한 결과를 반환한다
  - _Requirements: 2.3, 2.7, 2.8, 2.9_

- [ ] 4. user-account 인프라 계층 구현
- [ ] 4.1 Supabase Auth 기반 인증 리포지토리 구현체를 작성한다
  - 도메인 계층의 인증 리포지토리 인터페이스를 구현하여 Supabase Auth API를 호출한다
  - `signInWithPassword` 호출 결과를 인증 결과 타입으로 매핑한다
  - `signUp` 호출 시 반환된 사용자의 identities 배열이 비어 있으면 중복 이메일로 판별한다
  - `email_not_confirmed` 에러를 별도 분기하여 이메일 미인증 상태를 구분한다
  - 네트워크 오류 등 예상치 못한 에러를 `network_error` 또는 `unknown_error`로 매핑한다
  - `signOut`과 `verifyOtp` 메서드를 구현한다
  - 1단계에서 생성한 서버 클라이언트를 주입받아 사용한다
  - _Requirements: 1.3, 2.7, 4.2, 4.3, 5.3_

- [ ] 5. Next.js 미들웨어 및 인증 라우트 보호를 구성한다
  - 모든 요청에서 Supabase Auth 토큰을 갱신하는 미들웨어를 프로젝트 루트에 생성한다
  - `supabase.auth.getUser()`를 사용하여 JWT를 재검증한다(`getSession()` 사용 금지)
  - 인증된 사용자가 `/login` 또는 `/signup`에 접근하면 메인 페이지로 리다이렉트한다
  - 이메일 인증이 완료되지 않은 사용자가 보호된 페이지에 접근하면 접근을 차단하고 안내 메시지를 표시한다
  - 정적 자산(`_next/static`, `_next/image`, favicon, 이미지 파일)은 미들웨어 처리에서 제외한다
  - 미들웨어에서는 `request.cookies`와 `response.cookies`를 사용하여 토큰을 갱신한다
  - _Requirements: 4.1, 4.2, 4.4_

- [ ] 6. 서버 액션 및 이메일 인증 콜백 라우트 구현
- [ ] 6.1 로그인 서버 액션을 구현한다
  - 폼 데이터에서 이메일과 패스워드를 추출하여 로그인 유스케이스를 호출한다
  - 서버 클라이언트를 생성하고 인증 리포지토리 구현체를 조립하여 유스케이스에 주입한다
  - 성공 시 메인 페이지로 리다이렉트한다
  - 실패 시 인증 결과를 클라이언트에 반환한다
  - _Requirements: 1.3, 1.4_

- [ ] 6.2 회원가입 서버 액션을 구현한다
  - 폼 데이터에서 이메일과 패스워드를 추출하여 회원가입 유스케이스를 호출한다
  - 서버 클라이언트를 생성하고 인증 리포지토리 구현체를 조립하여 유스케이스에 주입한다
  - 성공 또는 실패 인증 결과를 클라이언트에 반환한다
  - _Requirements: 2.7_

- [ ] 6.3 (P) 로그아웃 서버 액션을 구현한다
  - 인증 리포지토리를 통해 세션을 종료하고 로그인 페이지로 리다이렉트한다
  - _Requirements: 4.3_

- [ ] 6.4 (P) 이메일 인증 콜백 라우트 핸들러를 구현한다
  - `token_hash`와 `type` 쿼리 파라미터를 추출하여 Supabase의 OTP 검증을 수행한다
  - 검증 성공 시 메인 페이지로, 실패 시 로그인 페이지로 리다이렉트한다
  - _Requirements: 2.7_

- [ ] 7. 인증 페이지 UI 구현
- [ ] 7.1 인증 페이지 전용 레이아웃과 공통 폼 상태 관리 훅을 구현한다
  - 인증 페이지(`/login`, `/signup`)에 적용되는 전용 레이아웃을 생성한다(사이드바, 헤더 없이 중앙 정렬 카드 형태)
  - `useActionState`를 래핑하여 서버 액션 호출 상태를 관리하는 커스텀 훅을 구현한다
  - 로딩 상태 추적, 중복 제출 방지, 네트워크 에러 감지 기능을 포함한다
  - _Requirements: 5.1, 5.2, 5.3_

- [ ] 7.2 로그인 폼 클라이언트 컴포넌트를 구현한다
  - 이메일 입력 필드, 패스워드 입력 필드, 로그인 버튼을 포함하는 폼을 구성한다
  - zod 스키마로 이메일 형식과 패스워드 빈 값을 클라이언트 사이드에서 검증한다
  - react-hook-form으로 폼 상태를 관리하고 shadcn/ui 컴포넌트로 UI를 렌더링한다
  - 서버 액션 결과에 따라 에러 메시지를 표시한다(잘못된 자격증명, 이메일 미인증, 미가입 이메일 안내)
  - 로딩 중 버튼을 비활성화하고 로딩 인디케이터를 표시한다
  - 네트워크 오류 시 재시도 옵션을 제공한다
  - "아직 계정이 없으신가요? 회원가입" 링크를 `/signup`으로 제공한다
  - _Requirements: 1.1, 1.2, 1.5, 1.6, 1.7, 1.8, 1.9, 3.1, 5.1, 5.3_

- [ ] 7.3 회원가입 폼 클라이언트 컴포넌트를 구현한다
  - 이메일 입력 필드, 패스워드 입력 필드, 패스워드 확인 입력 필드, 가입 버튼을 포함하는 폼을 구성한다
  - zod 스키마로 `@eluocnc.com` 도메인 제한, 이메일 형식, 패스워드 정책(8자 이상, 특수문자 포함), 패스워드 확인 일치를 클라이언트 사이드에서 검증한다
  - react-hook-form으로 폼 상태를 관리하고 shadcn/ui 컴포넌트로 UI를 렌더링한다
  - 가입 성공 시 "인증 메일이 발송되었습니다" 안내 화면으로 전환한다
  - 중복 이메일 시 이미 가입된 이메일임을 안내하고 로그인 페이지 링크를 제공한다
  - 로딩 중 버튼을 비활성화하고 로딩 인디케이터를 표시한다
  - 네트워크 오류 시 재시도 옵션을 제공한다
  - "이미 계정이 있으신가요? 로그인" 링크를 `/login`으로 제공한다
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.8, 2.9, 3.2, 5.2, 5.3_

- [ ] 8. 통합 검증 및 테스트
- [ ] 8.1 도메인 계층 단위 테스트를 작성한다
  - 이메일 값 객체의 유효한 eluocnc.com 이메일 생성, 타 도메인 거부, 잘못된 형식 거부를 검증한다
  - 패스워드 값 객체의 유효한 패스워드 생성, 8자 미만 거부, 특수문자 미포함 거부를 검증한다
  - _Requirements: 2.3, 2.4, 2.5_

- [ ] 8.2 애플리케이션 계층 단위 테스트를 작성한다
  - 로그인 유스케이스의 성공, 잘못된 자격증명, 이메일 미인증 시나리오별 인증 결과 반환을 검증한다
  - 회원가입 유스케이스의 성공, 중복 이메일, 도메인 위반 시나리오별 인증 결과 반환을 검증한다
  - 인증 리포지토리를 모킹하여 유스케이스 로직만 독립적으로 테스트한다
  - _Requirements: 1.3, 1.7, 1.8, 1.9, 2.3, 2.7, 2.8, 2.9_

- [ ] 8.3 인프라 계층 단위 테스트를 작성한다
  - Supabase Auth 에러를 도메인 에러 코드로 올바르게 매핑하는지 검증한다
  - 중복 이메일 감지 로직(identities 배열 확인)이 정상 동작하는지 검증한다
  - Supabase 클라이언트를 모킹하여 인프라 계층만 독립적으로 테스트한다
  - _Requirements: 1.3, 2.7, 5.3_

- [ ]*  8.4 UI 컴포넌트 테스트를 작성한다
  - 로그인 폼의 렌더링, zod 유효성 검증 에러 표시, 서버 에러 메시지 표시, 로딩 상태 UI를 검증한다
  - 회원가입 폼의 렌더링, 도메인 제한 에러, 패스워드 정책 에러, 가입 성공 안내 화면 전환을 검증한다
  - 로그인에서 회원가입, 회원가입에서 로그인 네비게이션 링크 동작을 검증한다
  - _Requirements: 1.2, 1.5, 1.6, 1.7, 1.8, 1.9, 2.2, 2.3, 2.4, 2.5, 2.6, 2.8, 2.9, 3.1, 3.2, 5.1, 5.2, 5.3_
