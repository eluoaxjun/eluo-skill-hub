# Feature Specification: 스킬 수정 팝업 디자인

**Feature Branch**: `013-skill-edit-popup-design`
**Created**: 2026-03-04
**Status**: Draft
**Input**: User description: "스킬 수정 팝업 디자인 — 수정버튼 클릭 시 팝업 표시, edit/[:id] 라우팅, 파일 삭제·교체 지원"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 스킬 카드에서 수정 팝업 열기 (Priority: P1)

관리자가 스킬 목록 화면에서 특정 스킬 카드의 수정 버튼을 클릭하면, 해당 스킬의 기존 데이터가 채워진 수정 팝업이 모달 형태로 나타난다. 팝업의 레이아웃과 구조는 기존 스킬 추가 팝업(2패널 레이아웃)과 동일하다.

**Why this priority**: 수정 기능의 핵심 진입점이자 가장 빈번한 사용 시나리오. 기존 추가 팝업 구조를 재사용하므로 일관된 UX를 제공한다.

**Independent Test**: 스킬 카드의 수정 버튼을 클릭하여 팝업이 열리고, 기존 데이터가 올바르게 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 관리자가 스킬 목록 화면에 있고 스킬이 1개 이상 존재할 때, **When** 스킬 카드의 수정 버튼을 클릭하면, **Then** 해당 스킬의 데이터가 미리 채워진 수정 팝업이 모달로 나타난다.
2. **Given** 수정 팝업이 열렸을 때, **When** 팝업의 각 필드를 확인하면, **Then** 아이콘, 제목, 설명, 카테고리, 공개 상태, 마크다운 파일명, 템플릿 파일 목록이 기존 데이터와 일치한다.
3. **Given** 수정 팝업이 열렸을 때, **When** Escape 키를 누르거나 X 버튼을 클릭하면, **Then** 변경사항이 없으면 즉시 닫히고, 변경사항이 있으면 닫기 확인 대화상자가 나타난다.

---

### User Story 2 - URL 직접 접근으로 수정 팝업 열기 (Priority: P1)

관리자가 브라우저 주소창에 `/admin/skills/edit/[id]`를 직접 입력하여 접근하면, 해당 스킬의 수정 화면이 표시된다. 인터셉팅 라우트로 접근하면 모달 형태로, 직접 URL로 접근하면 전체 페이지 형태로 표시된다.

**Why this priority**: URL 공유 및 북마크를 통한 직접 접근은 관리자 워크플로우의 필수 기능이며, 기존 추가 팝업의 라우팅 패턴과 동일한 방식으로 구현되어야 한다.

**Independent Test**: 브라우저 주소창에 유효한 스킬 ID의 edit URL을 입력하여 수정 화면이 올바르게 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 관리자가 스킬 목록 화면에서 수정 버튼을 클릭할 때(클라이언트 내비게이션), **When** `/admin/skills/edit/[id]`로 이동하면, **Then** 인터셉팅 라우트에 의해 모달 형태로 수정 팝업이 나타난다.
2. **Given** 관리자가 브라우저에 `/admin/skills/edit/[id]`를 직접 입력할 때, **When** 페이지가 로드되면, **Then** 전체 페이지 형태로 수정 화면이 표시된다.
3. **Given** 존재하지 않는 스킬 ID로 `/admin/skills/edit/[invalid-id]`에 접근할 때, **When** 페이지가 로드되면, **Then** 적절한 오류 메시지가 표시되고 스킬 목록으로 돌아갈 수 있는 링크가 제공된다.

---

### User Story 3 - 기존 파일 삭제 및 새 파일 업로드 (Priority: P2)

관리자가 수정 팝업에서 이미 업로드된 마크다운 파일이나 템플릿 파일을 삭제하고, 새로운 파일을 업로드할 수 있다. 파일 변경사항은 저장 버튼을 눌렀을 때 반영된다.

**Why this priority**: 스킬 내용 업데이트의 핵심 기능이지만, 팝업 열기(P1)가 먼저 구현되어야 한다.

**Independent Test**: 수정 팝업에서 기존 파일을 삭제하고 새 파일을 업로드한 후 저장하여, 변경사항이 올바르게 반영되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 수정 팝업에 마크다운 파일이 첨부되어 있을 때, **When** 삭제 버튼을 클릭하면, **Then** 파일이 제거되고 마크다운 미리보기 영역이 초기화된다.
2. **Given** 수정 팝업에서 마크다운 파일을 삭제한 후, **When** 새 파일을 업로드하면, **Then** 새 파일의 내용이 미리보기에 표시된다.
3. **Given** 수정 팝업에 템플릿 파일 3개가 첨부되어 있을 때, **When** 1개를 삭제하고 2개를 새로 추가하면, **Then** 총 4개의 템플릿 파일이 표시된다.
4. **Given** 파일을 변경한 후, **When** 저장 버튼을 클릭하면, **Then** 삭제된 파일은 스토리지에서 제거되고, 새 파일은 업로드되며, 데이터베이스가 업데이트된다.

---

### User Story 4 - 수정 내용 저장 (Priority: P2)

관리자가 수정 팝업에서 스킬 정보를 변경한 후 저장하면, 변경사항이 데이터베이스에 반영되고 스킬 목록이 갱신된다.

**Why this priority**: 수정 기능의 완결을 위해 필수이지만, 팝업 UI와 데이터 로드가 먼저 구현되어야 한다.

**Independent Test**: 수정 팝업에서 제목, 설명, 카테고리 등을 변경하고 저장한 후, 스킬 목록에서 변경사항이 반영되었는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 수정 팝업에서 제목을 변경했을 때, **When** 저장 버튼을 클릭하면, **Then** 변경된 제목이 저장되고 스킬 목록에 반영된다.
2. **Given** 수정 팝업에서 공개 상태를 변경했을 때, **When** 저장 버튼을 클릭하면, **Then** 상태가 업데이트되고 목록의 배지가 변경된다.
3. **Given** 수정 중 저장에 실패했을 때, **When** 오류가 발생하면, **Then** 오류 메시지가 표시되고 입력된 데이터는 유지된다.
4. **Given** 수정 팝업에서 아무것도 변경하지 않았을 때, **When** 저장 버튼을 클릭하면, **Then** 저장이 정상적으로 처리된다.

---

### Edge Cases

- 수정 팝업이 열려있는 동안 다른 관리자가 해당 스킬을 삭제한 경우: 저장 시 오류 메시지를 표시하고 목록으로 돌아간다.
- 파일 업로드 도중 네트워크 오류가 발생한 경우: 오류 메시지를 표시하고 재시도할 수 있도록 한다.
- 템플릿 파일을 전부 삭제하고 저장하는 경우: 파일 없이 저장이 정상 처리된다.
- 마크다운 파일만 삭제하고 저장하는 경우: 마크다운 없이 저장이 정상 처리된다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 스킬 카드의 수정 버튼 클릭 시 해당 스킬의 수정 팝업을 모달 형태로 표시해야 한다.
- **FR-002**: 수정 팝업은 스킬 추가 팝업과 동일한 2패널 레이아웃(좌측 메인 영역 + 우측 사이드바)을 사용해야 한다.
- **FR-003**: 수정 팝업은 `/admin/skills/edit/[id]` URL 패턴을 사용하며, 클라이언트 내비게이션 시 모달로, 직접 URL 접근 시 전체 페이지로 표시해야 한다.
- **FR-004**: 수정 팝업이 열릴 때 기존 스킬 데이터(아이콘, 제목, 설명, 카테고리, 공개 상태, 파일 정보)가 모든 필드에 미리 채워져야 한다.
- **FR-005**: 이미 업로드된 마크다운 파일의 이름과 미리보기가 표시되어야 하며, 삭제 버튼으로 제거할 수 있어야 한다.
- **FR-006**: 마크다운 파일 삭제 후 새 파일을 업로드할 수 있어야 한다.
- **FR-007**: 이미 업로드된 템플릿 파일 목록이 표시되어야 하며, 개별 삭제가 가능해야 한다.
- **FR-008**: 기존 템플릿 파일이 있는 상태에서 새 템플릿 파일을 추가할 수 있어야 하며, 총 파일 수는 10개를 초과할 수 없다.
- **FR-009**: 저장 시 변경된 필드만 업데이트하고, 삭제된 파일은 스토리지에서 제거하며, 새 파일은 업로드해야 한다.
- **FR-010**: 존재하지 않는 스킬 ID로 접근 시 오류 메시지를 표시하고 목록으로의 복귀 경로를 제공해야 한다.
- **FR-011**: 수정 팝업의 닫기 동작은 추가 팝업과 동일하게 변경사항 감지 및 닫기 확인 대화상자를 지원해야 한다.
- **FR-012**: 입력 필드의 유효성 검사 규칙은 추가 팝업과 동일해야 한다(제목 1-100자, 설명 1-500자, 카테고리 필수, 마크다운 1MB, 템플릿 100KB).

### Key Entities

- **Skill**: 수정 대상 스킬 — 아이콘, 제목, 설명, 카테고리, 공개 상태, 마크다운 파일, 템플릿 파일 목록을 포함
- **SkillFile (Markdown)**: 스킬의 상세 설명 마크다운 파일 — 기존 파일 참조(경로, 이름)와 신규 업로드 파일을 구분
- **SkillFile (Template)**: 스킬의 템플릿 파일 — 기존 파일 목록(경로, 이름, 크기)과 신규 업로드 파일을 구분

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 관리자가 수정 버튼 클릭 후 2초 이내에 기존 데이터가 채워진 수정 팝업을 확인할 수 있다.
- **SC-002**: 관리자가 스킬 수정(필드 변경 + 파일 교체 포함)을 3분 이내에 완료할 수 있다.
- **SC-003**: 수정 팝업의 모든 입력 필드와 파일 관리 기능이 추가 팝업과 동일한 UX를 제공한다.
- **SC-004**: URL 직접 접근 및 클라이언트 내비게이션 모두에서 수정 화면이 정상적으로 표시된다.
- **SC-005**: 파일 삭제·교체 작업이 저장 후 정확하게 반영되며, 불필요한 파일이 스토리지에 잔류하지 않는다.

## Assumptions

- 수정 팝업의 시각적 디자인(색상, 글래스모피즘, 라운딩 등)은 기존 추가 팝업과 완전히 동일하게 적용된다.
- 수정 팝업의 버튼 텍스트는 "스킬 저장하기" → "수정 저장하기" 등으로 컨텍스트에 맞게 변경된다.
- 임시저장 기능은 추가 팝업과 동일하게 수정 팝업에서도 지원된다.
- 한 번에 하나의 스킬만 수정할 수 있다(동시 수정 충돌 처리는 범위 밖).
- 스킬 수정 권한은 기존 스킬 추가 권한과 동일하게 관리자 역할로 제한된다.
