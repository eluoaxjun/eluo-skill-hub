## User Scenarios & Testing *(mandatory)*

### User Story 1 - 전체 서비스 현황 대시보드 조회 (Priority: P1)

관리자가 통계분석 페이지(`/admin/analytics`)에 접속하면, 서비스의 핵심 지표를 한눈에 파악할 수 있는 대시보드가 표시된다. 총 방문 수, 활성 사용자 수, 스킬 조회 수, 템플릿 다운로드 수 등 주요 지표가 요약 카드 형태로 상단에 노출되며, 일별 추이 차트가 함께 제공된다.

**Why this priority**: 관리자가 서비스 전반의 상태를 빠르게 파악하는 것이 통계분석의 가장 기본적인 가치이며, 이후 상세 분석의 진입점이 된다.

**Independent Test**: `/admin/analytics`에 접속하여 요약 카드의 숫자와 일별 추이 차트가 정상적으로 렌더링되는지 확인하면 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 관리자가 로그인한 상태, **When** `/admin/analytics`에 접속, **Then** 주요 지표 요약 카드(총 페이지뷰, 활성 사용자 수, 스킬 조회 수, 템플릿 다운로드 수)가 표시된다
2. **Given** 관리자가 통계분석 페이지에 있는 상태, **When** 기간 필터(오늘/최근 7일/최근 30일/커스텀)를 변경, **Then** 모든 지표와 차트가 선택한 기간에 맞게 갱신된다
3. **Given** 관리자가 통계분석 페이지에 있는 상태, **When** 페이지가 로드되면, **Then** 일별 이벤트 추이 차트(라인 차트)가 표시된다

---

### User Story 2 - 스킬별 인기도 분석 (Priority: P2)

관리자가 어떤 스킬이 가장 많이 조회되고 다운로드되는지 파악할 수 있다. 스킬 조회 순위, 템플릿 다운로드 순위, 북마크 추가/제거 현황을 테이블과 차트로 확인하여, 스킬 운영 전략을 수립하는 데 활용한다.

**Why this priority**: 스킬 허브의 핵심 콘텐츠인 스킬의 성과를 분석하는 것은 서비스 운영에 직접적인 인사이트를 제공하므로 두 번째 우선순위이다.

**Independent Test**: 통계분석 페이지에서 스킬별 분석 섹션에 진입하여 조회/다운로드 랭킹 테이블이 정상적으로 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 관리자가 통계분석 페이지에 있는 상태, **When** 스킬 분석 탭/섹션을 클릭, **Then** 스킬 조회수 상위 목록이 테이블로 표시된다
2. **Given** 스킬 분석 섹션이 표시된 상태, **When** 기간 필터를 적용, **Then** 해당 기간 내의 스킬 조회/다운로드 순위가 갱신된다
3. **Given** 스킬 분석 섹션이 표시된 상태, **When** 데이터가 로드되면, **Then** 스킬별 조회 수와 템플릿 다운로드 수를 함께 비교할 수 있는 형태로 표시된다

---

### User Story 3 - 사용자 행동 패턴 분석 (Priority: P3)

관리자가 사용자들의 네비게이션 패턴, 검색 행동, 인증 활동 등을 분석하여 UI/UX 개선 인사이트를 도출할 수 있다. 사이드바 클릭 분포, 페이지별 방문 현황, 검색 키워드 등의 데이터를 제공한다.

**Why this priority**: 사용자 행동 분석은 서비스 개선에 유용하지만, 기본 현황과 스킬 분석보다는 부가적인 성격이므로 세 번째 우선순위이다.

**Independent Test**: 사용자 행동 분석 섹션에서 사이드바 클릭 분포와 페이지별 방문 데이터가 정상적으로 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 관리자가 통계분석 페이지에 있는 상태, **When** 사용자 행동 탭/섹션을 클릭, **Then** 사이드바 탭별 클릭 분포(카테고리별 클릭 수)가 표시된다
2. **Given** 사용자 행동 섹션이 표시된 상태, **When** 데이터가 로드되면, **Then** 페이지별 방문 수 현황이 표시된다

---

### Edge Cases

- 선택한 기간 내에 이벤트 데이터가 없는 경우 빈 상태(empty state) 메시지를 표시한다
- event_logs 테이블에 데이터가 전혀 없는 초기 상태에서도 페이지가 오류 없이 렌더링되어야 한다
- 기간 필터에서 시작일이 종료일보다 이후인 경우 유효성 오류를 표시한다
- 대량의 이벤트 데이터(10만 건 이상)가 있어도 페이지 로딩 시간이 합리적이어야 한다
- properties 필드에 예상하지 못한 구조의 데이터가 들어온 경우에도 오류 없이 처리한다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 어드민 사이드바에 "통계분석" 메뉴 항목을 추가하여 `/admin/analytics` 경로로 접근할 수 있어야 한다
- **FR-002**: 시스템은 주요 지표 요약 카드(총 페이지뷰, 활성 사용자 수, 스킬 조회 수, 템플릿 다운로드 수)를 대시보드 상단에 표시해야 한다
- **FR-003**: 시스템은 기간 필터(오늘, 최근 7일, 최근 30일, 커스텀 날짜 범위)를 제공하여 모든 통계 데이터에 일괄 적용해야 한다
- **FR-004**: 시스템은 일별 이벤트 추이를 시각화하는 차트를 제공해야 한다
- **FR-005**: 시스템은 스킬별 조회 수 및 템플릿 다운로드 수 순위를 테이블로 표시해야 한다
- **FR-006**: 시스템은 사이드바 탭별 클릭 분포와 페이지별 방문 현황을 표시해야 한다
- **FR-007**: 시스템은 이전 기간 대비 증감률을 요약 카드에 함께 표시해야 한다 (예: 지난 7일 대비 +15%)
- **FR-008**: 시스템은 admin 권한을 가진 사용자만 통계분석 페이지에 접근할 수 있어야 한다 (기존 어드민 레이아웃의 권한 체계 활용)
- **FR-009**: 데이터가 없는 경우, 시스템은 적절한 빈 상태 메시지를 표시해야 한다

### Key Entities

- **이벤트 로그 (event_logs)**: 사용자 행동을 기록하는 핵심 데이터. 이벤트 이름, 사용자 ID, 세션 ID, 추가 속성(properties), 페이지 URL, 생성일시를 포함한다.
- **이벤트 유형**: 현재 기록되는 이벤트 카테고리 - 네비게이션(nav.page_view, nav.sidebar_click), 스킬 활동(skill.view, skill.template_download, skill.bookmark_add, skill.bookmark_remove), 검색(search.query), 인증(auth.signout)
- **이벤트 속성 (properties)**: 이벤트별 상세 정보를 담는 JSON 데이터. 페이지 경로(path), 스킬 ID(skill_id), 사이드바 탭(tab) 등 이벤트에 따라 다른 구조를 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 관리자가 통계분석 페이지에서 5초 이내에 전체 서비스 현황을 파악할 수 있다
- **SC-002**: 기간 필터 변경 후 3초 이내에 모든 지표와 차트가 갱신된다
- **SC-003**: 관리자가 인기 스킬 상위 10개를 별도의 수동 분석 없이 한 화면에서 확인할 수 있다
- **SC-004**: 통계분석 페이지의 모든 데이터가 event_logs 테이블의 실제 데이터와 일치한다 (정확성 100%)

## Assumptions

- 기존 어드민 레이아웃(`/admin/layout.tsx`)의 권한 체계(admin 역할만 접근)를 그대로 활용한다
- event_logs 테이블의 이벤트 유형은 현재 기록되는 8종(nav.page_view, nav.sidebar_click, skill.view, skill.template_download, skill.bookmark_add, skill.bookmark_remove, search.query, auth.signout)을 기준으로 하며, 향후 추가되는 이벤트도 자동으로 집계에 포함된다
- 차트 시각화는 기존 프로젝트에서 사용 중인 UI 라이브러리와 호환되는 방식으로 구현한다
- 기간 필터의 기본값은 "최근 7일"로 설정한다
- 통계 데이터 조회 시 서버 사이드에서 집계하여 성능을 보장한다
- 데이터 새로고침은 페이지 로드 또는 필터 변경 시 수행되며, 실시간 업데이트는 범위에 포함하지 않는다
