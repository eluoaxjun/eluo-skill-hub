## User Scenarios & Testing *(mandatory)*

### User Story 1 - 관리자가 회원의 역할(Role)을 변경한다 (Priority: P1)

관리자는 `/admin/members` 페이지에서 회원 목록을 확인하고, 각 회원의 역할을 드롭다운을 통해 변경할 수 있다. 역할 변경은 즉시 반영되며, 변경된 역할에 따라 해당 회원의 시스템 접근 권한이 달라진다.

**Why this priority**: 회원관리의 핵심 기능으로, 관리자가 조직 내 사용자 역할을 직접 통제할 수 있어야 한다. 이 기능 없이는 회원 권한 관리가 불가능하다.

**Independent Test**: 관리자 계정으로 로그인 후 `/admin/members` 페이지에 접속하여 특정 회원의 역할 드롭다운을 클릭하고 다른 역할을 선택했을 때, 변경이 성공적으로 저장되고 화면에 반영되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 관리자가 회원관리 페이지에 접속해 있고 회원 목록이 표시된 상태, **When** 특정 회원의 역할 드롭다운을 클릭하면, **Then** 사용 가능한 역할 목록(admin, user, viewer)이 표시된다
2. **Given** 역할 드롭다운이 열린 상태, **When** 새로운 역할을 선택하면, **Then** 해당 회원의 역할이 즉시 변경되고 성공 피드백이 표시된다
3. **Given** 관리자가 자기 자신의 역할을 변경하려 할 때, **When** 역할 드롭다운을 조작하면, **Then** 자기 자신의 역할은 변경할 수 없도록 드롭다운이 비활성화되어 있다
4. **Given** 역할 변경 요청이 실패한 경우, **When** 서버 오류가 발생하면, **Then** 오류 메시지가 표시되고 이전 역할 값이 유지된다

---

### User Story 2 - 회원 프로필에 이름이 표시된다 (Priority: P1)

회원 가입 시 입력한 이름이 프로필의 `name` 필드에 저장되고, 관리자 회원관리 페이지에서 각 회원의 이름이 표시된다. 기존 회원의 경우 `display_name` 데이터를 마이그레이션하여 `name` 필드를 채운다.

**Why this priority**: 이름 필드가 없으면 관리자가 이메일만으로 회원을 식별해야 하므로 관리 효율이 떨어진다. 역할 변경 기능과 함께 필수적이다.

**Independent Test**: 회원가입 후 관리자 페이지에서 해당 회원의 이름이 정상적으로 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 새로운 사용자가 회원가입 폼에서 이름을 입력하고 가입을 완료했을 때, **When** 관리자가 회원관리 페이지를 조회하면, **Then** 해당 회원의 이름이 목록에 표시된다
2. **Given** 기존 회원 중 `name` 필드가 비어 있는 회원이 존재할 때, **When** 데이터 마이그레이션이 실행되면, **Then** 기존에 저장된 `display_name` 값이 `name` 필드로 복사된다
3. **Given** 이름 없이 가입한 기존 회원이 있을 때, **When** 회원관리 페이지에서 조회하면, **Then** 이름 대신 이메일이 표시된다

---

### User Story 3 - 권한(Permission) 테이블 분리 및 역할-권한 매핑 (Priority: P2)

역할(Role)과 별도로 권한(Permission) 테이블을 분리하여, 회원이 수행할 수 있는 구체적인 행동(예: 스킬 파일 다운로드)을 관리한다. 각 역할에 기본 권한 세트가 매핑되며, 관리자는 추후 개별 회원의 권한을 조정할 수 있는 기반을 마련한다.

**Why this priority**: 현재 단계에서는 권한 테이블 구조와 역할-권한 매핑만 구축하며, 실제 다운로드 권한 제어는 추후 구현 예정이다. 데이터 모델 기반을 먼저 마련하는 것이 중요하다.

**Independent Test**: 권한 테이블이 생성되고 역할별 기본 권한이 올바르게 매핑되어 있는지 데이터베이스에서 확인한다.

**Acceptance Scenarios**:

1. **Given** 시스템에 권한 테이블이 존재할 때, **When** 역할별 기본 권한을 조회하면, **Then** admin과 user 역할에는 `skill_download` 권한이 포함되고, viewer 역할에는 포함되지 않는다
2. **Given** 새로운 회원이 가입하여 기본 역할(viewer)이 할당될 때, **When** 해당 회원의 권한을 조회하면, **Then** viewer 역할에 매핑된 기본 권한 세트가 반환된다
3. **Given** 관리자가 회원의 역할을 변경했을 때, **When** 변경된 역할에 매핑된 권한을 조회하면, **Then** 새로운 역할의 기본 권한 세트가 반환된다

---

### User Story 4 - 관리자가 회원을 검색한다 (Priority: P1)

관리자는 `/admin/members` 페이지 상단의 검색 입력 필드에 이름 또는 이메일을 입력하여 회원을 검색할 수 있다. 검색은 입력 즉시(debounce 300ms) 서버에서 수행되며, 검색어와 페이지 상태는 URL 파라미터(`?q=검색어&page=번호`)로 관리된다.

**Why this priority**: 회원 수가 많아질수록 전체 목록에서 특정 회원을 찾기 어려워지므로, 검색 기능은 회원관리의 기본 기능이다.

**Independent Test**: 관리자가 검색 필드에 이름 또는 이메일의 일부를 입력했을 때, 해당 검색어가 포함된 회원만 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 관리자가 회원관리 페이지에 접속해 있을 때, **When** 검색 입력 필드에 회원 이름의 일부를 입력하면, **Then** 300ms debounce 후 해당 이름이 포함된 회원만 필터링되어 표시된다
2. **Given** 관리자가 검색어를 입력한 상태, **When** 검색어를 이메일 주소의 일부로 변경하면, **Then** 해당 이메일이 포함된 회원만 필터링되어 표시된다
3. **Given** 검색 결과가 표시된 상태, **When** 검색어를 모두 지우면, **Then** 전체 회원 목록이 다시 표시된다
4. **Given** 검색어가 입력된 상태, **When** 브라우저 URL을 확인하면, **Then** `?q=검색어` 파라미터가 반영되어 있다
5. **Given** 검색 결과가 없을 때, **When** 일치하는 회원이 없으면, **Then** "검색 결과가 없습니다" 메시지가 표시된다

---

### User Story 5 - 회원 목록이 페이지네이션으로 표시된다 (Priority: P1)

관리자가 회원관리 페이지에서 회원 목록을 조회할 때, 페이지당 10명씩 표시되며 페이지 이동 UI를 통해 다음/이전 페이지로 이동할 수 있다. 검색 시에도 페이지네이션이 적용되며, 새 검색어 입력 시 1페이지로 리셋된다.

**Why this priority**: 회원 수 증가 시 전체 데이터를 한 번에 로드하면 성능 저하가 발생하므로, 서버사이드 페이지네이션은 필수이다.

**Independent Test**: 10명 이상의 회원이 존재할 때 페이지네이션 UI가 표시되고, 페이지 이동이 정상 동작하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 10명을 초과하는 회원이 존재할 때, **When** 회원관리 페이지를 조회하면, **Then** 페이지당 10명이 표시되고 페이지네이션 UI가 나타난다
2. **Given** 검색 결과가 여러 페이지에 걸칠 때, **When** 검색 후 2페이지로 이동하면, **Then** URL에 `?q=검색어&page=2`가 반영된다
3. **Given** 2페이지를 보고 있는 상태에서, **When** 새로운 검색어를 입력하면, **Then** 자동으로 1페이지로 리셋된다

---

### Edge Cases

- 동시에 여러 관리자가 같은 회원의 역할을 변경하려 할 때, 마지막 변경이 우선 적용된다 (Last Write Wins)
- 유일한 admin 역할 회원의 역할을 변경하려 할 때, 시스템에 최소 1명의 admin이 유지되어야 한다
- `name` 필드 마이그레이션 시 `display_name`이 null인 기존 회원은 `name`을 null로 유지하고 UI에서 이메일로 대체 표시한다
- 존재하지 않는 역할 ID로 변경 요청이 들어올 때 유효성 검사에서 거부한다
- 검색어 입력 중 빠르게 타이핑할 때 debounce(300ms)로 불필요한 서버 요청을 방지한다
- 검색 결과가 0건일 때 빈 상태 메시지("검색 결과가 없습니다")를 표시한다
- 검색어 변경 시 페이지를 1페이지로 자동 리셋한다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 관리자는 `/admin/members` 페이지에서 각 회원의 역할을 드롭다운 UI를 통해 변경할 수 있어야 한다
- **FR-002**: 역할 드롭다운에는 시스템에 정의된 모든 역할(admin, user, viewer)이 표시되어야 한다
- **FR-003**: 역할 변경은 선택 즉시 서버에 반영되어야 하며, 성공/실패 피드백을 사용자에게 제공해야 한다
- **FR-004**: 관리자는 자기 자신의 역할을 변경할 수 없어야 한다
- **FR-005**: 시스템에 최소 1명의 admin 역할 회원이 유지되어야 한다 (마지막 admin의 역할 변경 차단)
- **FR-006**: `profiles`에 `name` 필드가 추가되어야 하며, 회원가입 시 입력한 이름이 저장되어야 한다
- **FR-007**: 기존 회원의 `display_name` 값을 `profiles.name` 필드로 마이그레이션해야 한다
- **FR-008**: 회원관리 테이블에 이름 컬럼이 추가되어 표시되어야 하며, 이름이 없는 경우 이메일로 대체 표시해야 한다
- **FR-009**: 역할(Role)과 분리된 권한(Permission) 데이터가 존재해야 한다
- **FR-010**: 각 역할에 기본 권한 세트가 매핑되어야 한다 (역할-권한 매핑)
- **FR-011**: 회원의 권한은 해당 회원의 역할에 매핑된 권한을 통해 결정되어야 한다
- **FR-012**: 초기 권한으로 `skill_download`가 정의되어야 하며, admin/user 역할에 부여되고 viewer 역할에는 부여되지 않아야 한다
- **FR-013**: 관리자는 회원관리 페이지에서 이름 또는 이메일로 회원을 검색할 수 있어야 한다 (서버사이드 검색, debounce 300ms)
- **FR-014**: 검색어와 페이지 번호는 URL 파라미터(`?q=검색어&page=번호`)로 관리되어야 한다
- **FR-015**: 회원 목록은 페이지당 10명씩 서버사이드 페이지네이션으로 표시되어야 한다
- **FR-016**: 검색어 변경 시 페이지가 1페이지로 자동 리셋되어야 한다

### Key Entities

- **Profile (회원 프로필)**: 시스템 사용자의 기본 정보를 담는 엔티티. 이메일, 이름, 가입일, 역할(Role) 참조를 포함한다.
- **Role (역할)**: 사용자의 시스템 내 역할을 정의하는 엔티티. admin, user, viewer 등의 역할 이름을 가진다.
- **Permission (권한)**: 사용자가 수행할 수 있는 구체적인 행동을 정의하는 엔티티. 권한 이름(예: skill_download)과 설명을 가진다.
- **Role-Permission Mapping (역할-권한 매핑)**: 역할과 권한 간의 다대다 관계를 정의하는 엔티티. 각 역할에 어떤 권한이 부여되는지 매핑한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 관리자가 회원 역할을 3초 이내에 변경할 수 있다 (드롭다운 선택 → 변경 완료 피드백까지)
- **SC-002**: 회원관리 페이지에서 모든 회원의 이름이 정상적으로 표시된다 (이름이 없는 경우 이메일 대체)
- **SC-003**: 기존 회원 데이터의 이름 마이그레이션 성공률 100% (display_name이 존재하는 회원 기준)
- **SC-004**: 역할별 권한 조회 시 올바른 권한 세트가 반환된다 (admin/user: skill_download 포함, viewer: 미포함)
- **SC-005**: 관리자가 자기 자신의 역할 변경 시도 시 100% 차단된다
- **SC-006**: 회원 검색 결과가 debounce 포함 1초 이내에 표시된다
- **SC-007**: 100명 이상의 회원이 있을 때 페이지네이션이 정상 동작하며 페이지 로드가 2초 이내이다

## Assumptions

- 현재 시스템의 역할은 admin, user, viewer 3가지로 한정한다
- 권한의 실제 적용(스킬 다운로드 제어)은 본 스펙 범위 밖이며, 추후 별도 기능으로 구현된다
- 기존 회원의 기본 역할은 현재 할당된 역할을 유지하며, 신규 회원의 기본 역할은 viewer이다
- 역할-권한 매핑은 역할 단위로만 관리하며, 개별 회원별 권한 오버라이드는 본 스펙 범위 밖이다
- `/admin/members` URL 경로를 사용한다 (사용자 요청의 '/admin/memebers'는 오타로 판단)
