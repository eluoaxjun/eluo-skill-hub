# Feature Specification: 데이터베이스 쿼리 최적화 및 페이지 성능 개선

**Feature Branch**: `021-db-query-optimization`
**Created**: 2026-03-05
**Status**: Draft
**Input**: User description: "지금 전체 페이지가 너무 느려. 프로젝트 구조를 분석하고 성능을 파악해줘. database 쿼리 속도가 너무 느려."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 대시보드 페이지 빠른 로딩 (Priority: P1)

사용자가 대시보드(`/dashboard`)에 접속했을 때, 중복 인증 조회와 순차 쿼리로 인한 불필요한 대기 시간 없이 페이지가 빠르게 로딩된다. 현재 미들웨어, 레이아웃, 페이지에서 각각 독립적으로 인증을 호출하며, 북마크 조회가 인증 결과를 기다린 후 순차적으로 실행되어 최소 4회의 DB 라운드트립이 발생한다.

**Why this priority**: 대시보드는 가장 많은 사용자가 가장 자주 접근하는 페이지이며, 현재 중복 인증 호출(미들웨어 + 레이아웃 + 페이지)과 순차 쿼리로 인해 체감 속도가 가장 느린 페이지다.

**Independent Test**: 대시보드 페이지 로딩 시간을 측정하여 개선 전후 비교 가능. 대시보드만 단독으로 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 로그인된 사용자가 있을 때, **When** 대시보드 페이지에 접속하면, **Then** 인증 조회가 요청당 1회만 실행되고 페이지 로딩이 체감상 빨라진다
2. **Given** 대시보드 데이터가 로딩될 때, **When** 스킬 목록과 북마크 정보를 가져오면, **Then** 병렬로 가져올 수 있는 쿼리는 동시에 실행되어 대기 시간이 단축된다
3. **Given** 동일 요청 내에서, **When** 여러 컴포넌트가 인증 정보를 필요로 하면, **Then** 중복 호출 없이 캐싱된 인증 결과를 재사용한다

---

### User Story 2 - 어드민 스킬 편집 페이지 최적화 (Priority: P2)

관리자가 스킬 편집 페이지(`/admin/skills/edit/[id]`)에 접속했을 때, 현재 6회의 DB 라운드트립(인증 + 프로필/역할 + 인증 중복 + 역할 중복 + 스킬 상세 + 템플릿 순차)이 병렬화 및 중복 제거를 통해 최소화된다.

**Why this priority**: 관리자가 스킬을 수정할 때마다 접근하는 페이지로, 현재 가장 많은 순차 쿼리를 가진 페이지(6 라운드트립)이다. 스킬 상세와 템플릿 조회가 독립적인데도 순차 실행되고, 인증/역할 조회가 레이아웃과 페이지에서 중복된다.

**Independent Test**: 스킬 편집 페이지 로딩 시간 측정으로 단독 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 관리자가 스킬 편집 페이지에 접속할 때, **When** 스킬 상세와 템플릿 정보를 조회하면, **Then** 두 쿼리가 병렬로 실행되어 순차 대비 대기 시간이 줄어든다
2. **Given** 레이아웃에서 이미 인증 및 역할 확인을 했을 때, **When** 페이지 컴포넌트가 로딩되면, **Then** 동일 정보를 중복 조회하지 않고 재사용한다

---

### User Story 3 - 비효율 쿼리 패턴 개선 (Priority: P3)

시스템이 불필요한 데이터 조회(전체 행 가져온 후 카운팅, 이미 알고 있는 정보 재조회, 불필요한 JOIN으로 과다 데이터 전송)를 제거하여 DB 부하와 네트워크 전송량을 줄인다.

**Why this priority**: 개별 쿼리 최적화는 사용자에게 직접적인 체감 향상은 P1/P2보다 작지만, 데이터 규모 증가 시 성능 저하를 방지하고 서버 리소스를 절약한다.

**Independent Test**: 각 쿼리 개선을 독립적으로 적용하고 반환 데이터량 비교로 검증 가능.

**Acceptance Scenarios**:

1. **Given** 관리자 수를 확인할 때, **When** 관리자 카운트 조회를 실행하면, **Then** 전체 관리자 행을 가져오지 않고 카운트 값만 반환받는다
2. **Given** 카테고리 목록을 조회할 때, **When** 게시된 스킬이 있는 카테고리를 필터링하면, **Then** 각 카테고리에 속한 모든 스킬 ID를 가져오지 않고 존재 여부만 확인한다
3. **Given** 스킬을 수정할 때, **When** 마크다운 파일 경로가 필요하면, **Then** 이미 조회된 스킬 데이터에서 재사용하고 별도 쿼리를 실행하지 않는다

---

### User Story 4 - 피드백 및 응답 제출 시 병렬 처리 (Priority: P4)

사용자가 스킬에 피드백이나 답글을 작성할 때, 프로필 조회와 데이터 삽입이 병렬로 실행되고, 피드백 목록의 프로필 조회 라운드트립이 최소화된다.

**Why this priority**: 사용자 인터랙션 응답 속도에 직접 영향을 주지만 영향 범위가 피드백 기능에 한정된다.

**Independent Test**: 피드백 제출 응답 시간 측정으로 단독 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 사용자가 피드백을 제출할 때, **When** 피드백 저장과 프로필 정보 조회가 필요하면, **Then** 두 작업이 병렬로 실행된다
2. **Given** 피드백 목록을 로딩할 때, **When** 피드백 작성자와 답글 작성자 프로필을 조회하면, **Then** 가능한 한 적은 라운드트립으로 모든 프로필 정보를 가져온다

---

### Edge Cases

- 인증 캐시 적용 시 세션 만료 직후에도 캐싱된 인증 정보가 유효한 것처럼 처리되지 않아야 한다
- 병렬 쿼리 중 하나가 실패할 경우 나머지 쿼리 결과가 유실되지 않고 적절한 에러 처리가 이루어져야 한다
- 카운트 쿼리 최적화 시 RLS(Row Level Security) 정책이 동일하게 적용되어야 한다
- 대규모 데이터(수백 개 스킬, 수천 개 피드백) 환경에서도 최적화된 쿼리가 타임아웃 없이 실행되어야 한다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 동일 요청 내 중복 인증 조회를 제거하고 요청당 최대 1회 인증 확인만 수행해야 한다
- **FR-002**: 시스템은 데이터 의존성이 없는 독립적인 DB 쿼리를 병렬로 실행해야 한다
- **FR-003**: 시스템은 스킬 상세 조회에서 스킬 정보와 템플릿 목록을 병렬로 가져와야 한다
- **FR-004**: 시스템은 회원 역할 변경 시 역할 목록 조회와 현재 역할 조회를 병렬로 실행해야 한다
- **FR-005**: 시스템은 관리자 수 카운팅 시 전체 행을 가져오지 않고 DB 수준 카운트를 사용해야 한다
- **FR-006**: 시스템은 스킬 수정 시 이미 조회된 마크다운 파일 경로를 재사용하고 별도 쿼리를 실행하지 않아야 한다
- **FR-007**: 시스템은 피드백/답글 제출 시 데이터 삽입과 프로필 조회를 병렬로 실행해야 한다
- **FR-008**: 시스템은 카테고리 조회 시 과다 JOIN 없이 효율적으로 필터링해야 한다
- **FR-009**: 시스템은 어드민 스킬 편집 페이지에서 레이아웃과 중복되는 인증/역할 조회를 제거해야 한다
- **FR-010**: 시스템은 템플릿 다운로드 시 템플릿 조회와 역할 확인을 병렬로 실행해야 한다

### Key Entities

- **Skills**: 자동화 스킬 정보. 카테고리, 템플릿과 관계 보유
- **Profiles**: 사용자 프로필. 역할(roles)과 관계 보유
- **Skill Templates**: 스킬별 템플릿 파일. 스킬과 N:1 관계
- **Skill Feedback Logs**: 스킬에 대한 피드백. 사용자 프로필과 관계 보유
- **Feedback Replies**: 피드백에 대한 답글. 피드백 및 사용자 프로필과 관계 보유
- **Categories**: 스킬 분류. 스킬과 1:N 관계
- **Bookmarks**: 사용자별 스킬 북마크. 사용자와 스킬의 M:N 관계

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 대시보드 페이지 초기 로딩 시 DB 라운드트립이 4회에서 2회 이하로 감소한다
- **SC-002**: 어드민 스킬 편집 페이지 초기 로딩 시 DB 라운드트립이 6회에서 3회 이하로 감소한다
- **SC-003**: 피드백 제출 응답 시 DB 라운드트립이 2회에서 1회로 감소한다
- **SC-004**: 관리자 카운트 조회 시 네트워크 전송 데이터량이 기존 대비 90% 이상 감소한다
- **SC-005**: 모든 최적화 적용 후 기존 기능의 정상 동작이 유지된다 (회귀 버그 0건)
- **SC-006**: 최적화 적용 후 사용자 체감 페이지 로딩 속도가 개선된다

## Assumptions

- 인증 라이브러리가 요청당 결과를 자동 캐싱하지 않으며, 별도 캐싱 래퍼 적용이 필요하다
- 병렬 쿼리 실행이 DB 커넥션 풀 한도 내에서 안전하게 동작한다
- 카테고리 필터링 최적화를 위해 DB 뷰 또는 RPC 함수 생성이 가능하다
- RLS 정책은 카운트 쿼리 모드에서도 동일하게 적용된다
- 현재 클라이언트 사이드 수동 캐싱 패턴(스킬 상세 모달)은 이번 최적화 범위에서 유지한다
