# Feature Specification: 스킬 관리 페이지 테이블뷰 추가

**Feature Branch**: `026-admin-skill-table-view`
**Created**: 2026-03-10
**Status**: Draft
**Input**: User description: "스킬관리페이지 테이블뷰 형식 추가. 검색창 왼쪽에 토글버튼(그리드뷰, 테이블뷰) 추가. default는 그리드. 테이블뷰 아이콘을 클릭하면 데이터들이 테이블 뷰처럼 보여야한다. 클릭했을때 동작은 동일하게."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 뷰 모드 토글 (Priority: P1)

관리자가 스킬 관리 페이지에서 검색창 왼쪽의 토글 버튼을 클릭하여 그리드뷰와 테이블뷰를 전환할 수 있다. 기본 뷰 모드는 그리드뷰이며, 토글 시 동일한 스킬 데이터가 선택한 뷰 형식으로 즉시 전환된다.

**Why this priority**: 핵심 기능으로, 이 토글이 없으면 테이블뷰 자체에 접근할 수 없다.

**Independent Test**: 스킬 관리 페이지 진입 후 토글 버튼을 클릭하여 뷰가 전환되는지 확인.

**Acceptance Scenarios**:

1. **Given** 관리자가 스킬 관리 페이지에 진입한 상태, **When** 페이지가 로드됨, **Then** 그리드뷰가 기본으로 표시되고 그리드 아이콘이 활성 상태로 보인다.
2. **Given** 그리드뷰가 활성화된 상태, **When** 테이블뷰 아이콘을 클릭, **Then** 동일한 스킬 목록이 테이블 형식으로 전환되어 표시된다.
3. **Given** 테이블뷰가 활성화된 상태, **When** 그리드뷰 아이콘을 클릭, **Then** 동일한 스킬 목록이 기존 카드 그리드 형식으로 복원된다.

---

### User Story 2 - 테이블뷰에서 스킬 데이터 표시 (Priority: P1)

관리자가 테이블뷰에서 각 스킬의 주요 정보(제목, 카테고리, 버전, 상태, 태그, 생성일, 수정일)를 행 단위로 한눈에 확인할 수 있다.

**Why this priority**: 테이블뷰의 핵심 가치로, 많은 스킬을 빠르게 스캔하고 비교할 수 있어야 한다.

**Independent Test**: 테이블뷰로 전환 후 각 컬럼에 올바른 데이터가 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 테이블뷰가 활성화된 상태, **When** 스킬 목록을 확인, **Then** 각 행에 제목, 카테고리, 버전, 상태(Published/Draft), 태그, 생성일, 수정일이 표시된다.
2. **Given** 테이블뷰에 스킬이 표시된 상태, **When** 상태 값을 확인, **Then** Published는 초록색 뱃지, Draft는 회색 뱃지로 시각적으로 구분된다.

---

### User Story 3 - 테이블뷰에서 스킬 수정/삭제 동작 (Priority: P1)

관리자가 테이블뷰에서도 그리드뷰와 동일하게 각 스킬에 대해 수정 및 삭제 동작을 수행할 수 있다.

**Why this priority**: 뷰 전환 시 기존 동작이 그대로 유지되어야 사용성이 보장된다.

**Independent Test**: 테이블뷰에서 수정/삭제 버튼을 클릭하여 동작이 그리드뷰와 동일한지 확인.

**Acceptance Scenarios**:

1. **Given** 테이블뷰에 스킬이 표시된 상태, **When** 특정 스킬의 수정 버튼을 클릭, **Then** 해당 스킬의 수정 페이지로 이동한다.
2. **Given** 테이블뷰에 스킬이 표시된 상태, **When** 특정 스킬의 삭제 버튼을 클릭, **Then** 삭제 확인 다이얼로그가 표시된다.

---

### User Story 4 - 뷰 모드와 필터/검색/페이지네이션 연동 (Priority: P2)

관리자가 뷰 모드를 전환해도 현재 적용된 검색어, 필터(상태, 카테고리), 페이지네이션 상태가 유지된다.

**Why this priority**: 필터 상태 유지가 안 되면 뷰 전환 시마다 검색을 다시 해야 하므로 사용성이 크게 저하된다.

**Independent Test**: 검색/필터 적용 후 뷰 모드를 전환하여 결과가 유지되는지 확인.

**Acceptance Scenarios**:

1. **Given** 그리드뷰에서 "기획" 카테고리로 필터링된 상태, **When** 테이블뷰로 전환, **Then** 동일한 "기획" 카테고리 필터가 적용된 결과가 테이블 형식으로 표시된다.
2. **Given** 테이블뷰에서 검색어를 입력한 상태, **When** 그리드뷰로 전환, **Then** 동일한 검색 결과가 카드 형식으로 표시된다.

---

### Edge Cases

- 스킬이 0개일 때 테이블뷰에서도 "등록된 스킬이 없습니다" 빈 상태 메시지가 적절히 표시되는가?
- 테이블뷰에서 태그가 많은 스킬의 경우 태그 표시가 행 높이를 과도하게 늘리지 않는가?
- 좁은 화면에서 테이블뷰가 가로 스크롤로 대응되는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 검색창 왼쪽에 그리드뷰/테이블뷰 토글 버튼이 표시되어야 한다.
- **FR-002**: 토글 버튼은 그리드 아이콘과 테이블(리스트) 아이콘 두 개로 구성되며, 현재 활성 뷰가 시각적으로 구분되어야 한다.
- **FR-003**: 페이지 진입 시 기본 뷰 모드는 그리드뷰여야 한다.
- **FR-004**: 테이블뷰에서 각 스킬은 행 단위로 표시되며, 컬럼은 제목, 카테고리, 버전, 상태, 태그, 생성일, 수정일을 포함해야 한다.
- **FR-005**: 테이블뷰의 각 행에서 수정/삭제 동작이 그리드뷰와 동일하게 작동해야 한다.
- **FR-006**: 뷰 모드 전환 시 현재 적용된 검색어, 필터, 페이지네이션 상태가 유지되어야 한다.
- **FR-007**: 뷰 모드 전환은 페이지 새로고침 없이 즉시 이루어져야 한다.
- **FR-008**: 스킬이 0개인 경우 테이블뷰에서도 적절한 빈 상태 메시지가 표시되어야 한다.
- **FR-009**: 테이블뷰에서 태그는 최대 3개까지 표시하고, 초과 시 "+N" 형태로 축약 표시해야 한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 관리자가 토글 버튼 클릭 1회로 뷰 모드를 즉시 전환할 수 있다.
- **SC-002**: 테이블뷰에서 10개 이상의 스킬을 한 화면에서 스캔할 수 있어 그리드뷰 대비 정보 밀도가 향상된다.
- **SC-003**: 뷰 전환 시 기존 필터/검색 상태가 100% 유지된다.
- **SC-004**: 테이블뷰에서의 수정/삭제 동작 성공률이 그리드뷰와 동일하다.

## Assumptions

- 뷰 모드 상태는 클라이언트 측에서만 관리하며, 페이지 새로고침 시 기본값(그리드뷰)으로 초기화된다.
- 테이블뷰의 컬럼 정렬 기능은 본 스펙 범위에 포함하지 않는다(기존 서버 정렬 유지).
- 테이블뷰는 데스크탑 환경을 기본 대상으로 하며, 좁은 화면에서는 가로 스크롤로 대응한다.
