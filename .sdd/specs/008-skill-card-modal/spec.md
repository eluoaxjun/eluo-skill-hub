# Feature Spec: 스킬 카드 팝업 (모달, 북마크, 피드백)

> **Feature ID**: 008
> **Status**: Draft
> **Created**: 2026-03-02
> **Depends on**: 001-root-page, 002-auth, 003-sidebar-refactor, 005-admin-skill-management

---

## 1. 기능 개요

대시보드(추천 에이전트 섹션)에 실제 DB 데이터를 기반으로 스킬 카드 그리드를 표시하고,
카드 클릭 시 상세 모달이 열리도록 한다.

모달에서는 스킬 상세 정보 확인, 북마크 등록/해제, 피드백(별점+코멘트) 제출이 가능하다.

북마크된 스킬은 '내 에이전트' 탭(`/myagent`)에서 동일한 카드 그리드 형식으로 확인할 수 있다.

---

## 2. 유저 스토리

### US-001: 추천 스킬 탐색
> As a 로그인 사용자,
> I want to 대시보드에서 DB에 등록된 스킬들을 카드 그리드로 탐색하기를,
> So that 어떤 스킬이 있는지 한눈에 파악할 수 있다.

### US-002: 스킬 상세 모달 확인
> As a 로그인 사용자,
> I want to 스킬 카드를 클릭하면 모달로 상세 정보를 확인하기를,
> So that 페이지 이동 없이 스킬의 설명, 사용 방법, 피드백을 빠르게 확인할 수 있다.

### US-003: 스킬 북마크
> As a 로그인 사용자,
> I want to 모달에서 스킬을 북마크/해제하기를,
> So that 내가 즐겨 사용하는 스킬을 '내 에이전트' 탭에서 빠르게 접근할 수 있다.

### US-004: 내 에이전트 탭에서 북마크 확인
> As a 로그인 사용자,
> I want to '내 에이전트' 탭에서 북마크한 스킬 목록을 카드 그리드로 보기를,
> So that 내가 저장한 스킬들을 한 곳에서 관리할 수 있다.

### US-005: 스킬 피드백 제출
> As a 로그인 사용자,
> I want to 모달에서 별점과 코멘트로 피드백을 제출하기를,
> So that 스킬 품질 개선에 기여할 수 있다.

## 3. 기능 요구사항

### 3.1 대시보드 스킬 카드 그리드 (DB 연동)

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-01 | 스킬 목록은 `skills` 테이블에서 조회한다 (기존 InMemorySkillRepository 제거) | 필수 |
| FR-02 | `status = 'active'`인 스킬만 표시한다 | 필수 |
| FR-03 | 카드에 표시할 아이콘은 사전 정의된 이모지 목록에서 `skill.id` 기반으로 결정적(deterministic) 랜덤 할당한다 | 필수 |
| FR-04 | 각 카드에 스킬 제목, 설명, 카테고리 태그를 표시한다 | 필수 |
| FR-05 | 카테고리 필터(URL 쿼리 파라미터 `?category=slug`)가 적용되어 있으면 해당 카테고리의 스킬만 표시한다 | 필수 |
| FR-06 | "전체 보기" (`/marketplace`) 링크를 제거한다 | 필수 |
| FR-07 | 스킬 데이터 로딩 중 스켈레톤(skeleton) 상태를 표시한다 | 권장 |

### 3.2 스킬 상세 모달

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-08 | 스킬 카드 클릭 시 모달이 열린다 (페이지 이동 없음) | 필수 |
| FR-09 | 모달은 배경 오버레이(backdrop)와 함께 화면 중앙에 표시된다 | 필수 |
| FR-10 | 모달 닫기: 닫기(X) 버튼 클릭 또는 배경 클릭 또는 ESC 키로 닫을 수 있다 | 필수 |
| FR-11 | 모달 좌측 영역에 스킬 아이콘, 제목, 카테고리 태그, 제작자 정보를 표시한다 | 필수 |
| FR-12 | 모달 좌측 영역에 스킬의 `markdown_content`를 렌더링하여 표시한다 | 필수 |
| FR-13 | 모달 우측 사이드 패널에 북마크 버튼, 클립보드 복사 버튼을 표시한다 | 필수 |
| FR-14 | 모달 우측 사이드 패널에 스킬 메타데이터(카테고리, 최근 업데이트)를 표시한다 | 필수 |
| FR-15 | 모달 좌측 하단에 피드백 섹션(별점 + 코멘트 + 제출 버튼)을 표시한다 | 필수 |
| FR-16 | 모달 레이아웃은 `reference/skill-modal.html`의 2단(좌: 스크롤 가능 콘텐츠, 우: 고정 사이드 패널) 구조를 따른다 | 필수 |

### 3.3 북마크 기능

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-17 | 로그인 사용자는 모달 내 북마크 버튼으로 스킬을 북마크/해제할 수 있다 | 필수 |
| FR-18 | 북마크 상태는 `bookmarks` 테이블에 저장한다 (user_id, skill_id 복합 유니크) | 필수 |
| FR-19 | 북마크 버튼은 현재 북마크 상태를 시각적으로 표시한다 (채움/비채움 아이콘) | 필수 |
| FR-20 | 스킬 카드 그리드에서도 현재 북마크 상태를 카드 상단에 표시한다 (북마크 아이콘 뱃지) | 권장 |

### 3.4 내 에이전트 페이지 (북마크 목록)

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-21 | `/myagent` 페이지에서 현재 로그인 사용자가 북마크한 스킬을 카드 그리드로 표시한다 | 필수 |
| FR-22 | 북마크한 스킬이 없으면 "아직 북마크한 에이전트가 없습니다" 빈 상태를 표시한다 | 필수 |
| FR-23 | 북마크된 스킬 카드도 대시보드와 동일한 `SkillCard` 컴포넌트를 재사용한다 | 필수 |
| FR-24 | 내 에이전트 페이지의 카드도 클릭 시 동일한 스킬 상세 모달이 열린다 | 필수 |

### 3.5 피드백 기능

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-25 | 모달 피드백 섹션에서 별점(1~5점)과 코멘트를 입력하고 제출할 수 있다 | 필수 |
| FR-26 | 제출된 피드백은 `skill_feedback_logs` 테이블에 저장한다 | 필수 |
| FR-27 | 피드백 제출 후 성공 메시지를 표시한다 | 필수 |
| FR-28 | 피드백 데이터는 대시보드 어디에도 표시하지 않는다 (로그 목적만) | 필수 |
| FR-29 | 동일 사용자가 동일 스킬에 여러 번 피드백을 제출할 수 있다 (누적 로그) | 권장 |

### 3.6 DB 스키마 추가

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-30 | `bookmarks` 테이블 생성: id, user_id(FK→auth.users), skill_id(FK→skills), created_at | 필수 |
| FR-31 | `bookmarks` 테이블에 (user_id, skill_id) 복합 유니크 제약 추가 | 필수 |
| FR-32 | `skill_feedback_logs` 테이블 생성: id, user_id(FK→auth.users), skill_id(FK→skills), rating(1~5), comment, created_at | 필수 |
| FR-33 | 두 테이블에 RLS 정책 적용 (자기 데이터만 읽기/쓰기) | 필수 |

## 5. 가정 사항

1. `skills` 테이블에 `status = 'active'`인 스킬 데이터가 존재한다
2. `skills` 테이블에 icon 필드가 없으므로, 아이콘은 프론트엔드에서 skill.id 해시 기반으로 이모지를 결정적 랜덤 할당한다
3. 기존 `Skill` 도메인 엔티티와 `InMemorySkillRepository`는 이 피처에서 대체되지만, 관련 테스트는 유지된다
4. 피드백 조회/관리 UI는 이 명세 범위에 포함하지 않는다
5. 스킬의 `markdown_content`는 마크다운 형식으로 저장되어 있다고 가정한다
6. 003-sidebar-refactor에서 정의된 `/marketplace` 라우트 제거와 이 명세의 "전체 보기 링크 제거"는 동일 맥락이다

## 6. 제약 조건

1. Constitution DDD 아키텍처 준수 (domain → application → infrastructure 계층 분리)
2. `any` 타입 사용 금지
3. TypeScript strict 모드 준수
4. 외부 UI 라이브러리 추가 없이 Tailwind CSS로 구현 (reference/skill-modal.html 참조)
5. 모달 상태 관리는 클라이언트 컴포넌트에서 처리하되, 데이터 페칭은 서버 컴포넌트 또는 Server Action으로 처리한다
6. 북마크/피드백 제출은 Next.js Server Action 또는 API Route로 처리한다

## 7. 수용 기준 체크리스트

### 스킬 카드 그리드
- [ ] 대시보드에 DB `skills` 테이블의 `status='active'` 스킬이 표시된다
- [ ] 각 스킬 카드에 아이콘(이모지), 제목, 설명, 카테고리 태그가 표시된다
- [ ] 아이콘은 skill.id 기반으로 결정적으로 배정된다 (페이지 새로고침 후에도 동일)
- [ ] 카테고리 필터 적용 시 해당 카테고리 스킬만 표시된다
- [ ] "전체 보기" 링크가 제거되었다

### 스킬 상세 모달
- [ ] 카드 클릭 시 모달이 열린다 (페이지 이동 없음)
- [ ] 모달에 스킬 제목, 카테고리, 설명(markdown_content)이 표시된다
- [ ] X 버튼, 배경 클릭, ESC 키로 모달을 닫을 수 있다
- [ ] 모달 레이아웃이 reference/skill-modal.html의 2단 구조를 따른다
- [ ] 다크 모드에서 모달이 정상 표시된다

### 북마크
- [ ] 모달 내 북마크 버튼을 클릭하면 북마크가 토글된다
- [ ] 북마크 상태가 DB에 저장되고 페이지 새로고침 후에도 유지된다
- [ ] `/myagent` 페이지에서 북마크한 스킬이 카드 그리드로 표시된다
- [ ] 북마크가 없으면 빈 상태 메시지가 표시된다

### 피드백
- [ ] 모달 내 별점(1~5)과 코멘트를 입력하고 제출할 수 있다
- [ ] 제출 후 성공 메시지가 표시된다
- [ ] 피드백이 `skill_feedback_logs` 테이블에 저장된다

### DB 스키마
- [ ] `bookmarks` 테이블이 생성되었다
- [ ] `bookmarks`에 (user_id, skill_id) 복합 유니크 제약이 적용되었다
- [ ] `skill_feedback_logs` 테이블이 생성되었다
- [ ] 두 테이블에 RLS 정책이 적용되었다

## 부록: UI 레이아웃 참조

### 스킬 카드 (기존과 유사, 북마크 뱃지 추가)
```
┌────────────────────────────┐
│ 🤖  (결정적 랜덤 이모지)   [🔖] │
│ 스킬 제목                   │
│ 스킬 설명 (2줄 말줄임...)    │
│ [카테고리]                  │
└────────────────────────────┘
```

### 스킬 상세 모달 (reference/skill-modal.html 기준)
```
┌──────────────────────────────────────────────────────┐
│  [X 닫기]                                             │
│  ┌────────────────────────────┬──────────────────┐   │
│  │ 🖋️ 스킬 제목                │ [액션 패널]       │   │
│  │ 카테고리 • 버전              │                  │   │
│  │                            │ [북마크 버튼]     │   │
│  │ ── 상세 설명 ──             │ [클립보드 복사]   │   │
│  │ markdown_content 렌더링    │                  │   │
│  │                            │ ── 스킬 정보 ──  │   │
│  │ ── 피드백 및 리뷰 ──        │ 카테고리         │   │
│  │ [★★★★☆] 별점 선택          │ 최근 업데이트    │   │
│  │ [코멘트 입력창]             │                  │   │
│  │ [피드백 제출 버튼]          │                  │   │
│  └────────────────────────────┴──────────────────┘   │
└──────────────────────────────────────────────────────┘
```

### 도메인 이벤트 흐름
```
사용자 북마크 클릭
  → BookmarkToggledEvent 발생
  → BookmarkRepository.toggle(userId, skillId)
  → UI 낙관적 업데이트

사용자 피드백 제출
  → FeedbackSubmittedEvent 발생
  → FeedbackLogRepository.save(log)
  → 성공 메시지 표시
```
