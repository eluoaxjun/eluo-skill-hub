# Feature Spec: 사이드바 리팩토링 및 카테고리 DB 관리

> **Feature ID**: 003
> **Status**: Draft
> **Created**: 2026-03-02
> **Depends on**: 001-root-page, 002-auth

---

## 1. 기능 개요

기존 사이드바의 메뉴 구조를 변경하고, 카테고리를 DB 테이블로 관리하도록 개선한다.

**주요 변경:**
- 메인 메뉴에서 "마켓플레이스" 제거
- 카테고리를 기획/디자인/퍼블리싱/개발/QA 5개로 재정의
- 카테고리를 하드코딩이 아닌 DB 테이블에서 조회
- "내 에이전트" 메뉴 클릭 시 `/myagent` 경로로 이동하되, 동일한 레이아웃(사이드바+헤더) 유지
- 카테고리 클릭 시 대시보드에서 해당 카테고리로 스킬 목록 필터링

---

## 2. 유저 스토리

### US-001: 사이드바 메뉴 탐색
> As a 로그인 사용자,
> I want to 사이드바에서 대시보드, 내 에이전트, 카테고리 메뉴를 통해 원하는 영역으로 이동하기를,
> So that 불필요한 메뉴 없이 핵심 기능에 빠르게 접근할 수 있다.

### US-002: 카테고리별 스킬 필터링
> As a 로그인 사용자,
> I want to 사이드바에서 카테고리(기획, 디자인, 퍼블리싱, 개발, QA)를 클릭하여 해당 카테고리의 스킬만 보기를,
> So that 내 직군에 맞는 스킬을 빠르게 찾을 수 있다.

### US-003: 내 에이전트 페이지 접근
> As a 로그인 사용자,
> I want to "내 에이전트" 메뉴를 클릭하면 `/myagent` 페이지로 이동하기를,
> So that 내가 북마크한 스킬 목록을 확인할 수 있다.

### US-004: 카테고리 관리 (관리자)
> As a 관리자,
> I want to 카테고리를 DB에서 추가/수정/삭제할 수 있기를,
> So that 조직의 직군 구조가 변경되어도 유연하게 대응할 수 있다.

## 3. 기능 요구사항

### 3.1 사이드바 메뉴 구조 변경

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-01 | "메인" 그룹에 "대시보드"와 "내 에이전트" 두 항목만 표시한다 (마켓플레이스 제거) | 필수 |
| FR-02 | "내 에이전트" 클릭 시 `/myagent` 경로로 이동한다 | 필수 |
| FR-03 | `/myagent` 페이지는 루트 페이지와 동일한 레이아웃(사이드바+헤더)을 유지한다 | 필수 |
| FR-04 | 현재 선택된 메뉴 항목을 시각적으로 활성 상태로 표시한다 | 필수 |

### 3.2 카테고리 메뉴

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-05 | "카테고리" 그룹에 기획, 디자인, 퍼블리싱, 개발, QA 5개 항목을 표시한다 | 필수 |
| FR-06 | 카테고리 목록은 DB 테이블에서 조회하여 동적으로 렌더링한다 | 필수 |
| FR-07 | 카테고리 클릭 시 대시보드(`/`)에서 해당 카테고리로 스킬 목록을 필터링한다 | 필수 |
| FR-08 | 카테고리 필터링 상태를 URL 쿼리 파라미터로 관리한다 (예: `/?category=planning`) | 필수 |
| FR-09 | 필터링된 카테고리에 해당하는 사이드바 항목을 활성 상태로 표시한다 | 필수 |
| FR-10 | 카테고리 데이터 로딩 중에는 사이드바에 스켈레톤/로딩 상태를 표시한다 | 권장 |

### 3.3 카테고리 DB 테이블

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-11 | `categories` 테이블을 생성한다 (id, name, slug, icon, sort_order, created_at, updated_at) | 필수 |
| FR-12 | 초기 시드 데이터로 5개 카테고리(기획, 디자인, 퍼블리싱, 개발, QA)를 삽입한다 | 필수 |
| FR-13 | 카테고리 테이블에 대해 관리자 CRUD(추가/수정/삭제)가 가능하도록 설계한다 | 필수 |
| FR-14 | `sort_order` 필드를 통해 사이드바 표시 순서를 제어한다 | 필수 |

### 3.4 내 에이전트 페이지

| # | 요구사항 | 우선순위 |
|---|---------|---------|
| FR-15 | `/myagent` 경로에 페이지를 생성한다 | 필수 |
| FR-16 | 페이지 콘텐츠는 빈 상태(placeholder)로 구현한다 (북마크 기능은 추후 별도 명세) | 필수 |
| FR-17 | "아직 북마크한 에이전트가 없습니다" 등의 빈 상태 메시지를 표시한다 | 권장 |

## 4. 비기능 요구사항

| # | 요구사항 | 기준 |
|---|---------|------|
| NFR-01 | 카테고리 조회 API 응답 시간은 500ms 이내여야 한다 | Constitution §8 |
| NFR-02 | 사이드바 카테고리 목록은 서버 사이드에서 렌더링하여 FOUC를 방지한다 | UX 품질 |
| NFR-03 | 다크 모드에서 모든 변경 사항이 정상 동작해야 한다 | 001-root-page NFR-02 |
| NFR-04 | 카테고리 테이블에 RLS(Row Level Security) 정책을 적용한다 (읽기: 모든 인증 사용자, 쓰기: 관리자만) | Constitution §7 |

---

## 5. 가정 사항

1. 002-auth 인증 시스템이 구현되어 있다
2. 001-root-page의 대시보드 레이아웃(사이드바+헤더+메인 콘텐츠)이 구현되어 있다
3. 관리자 CRUD UI는 이 명세 범위에 포함하지 않으며, DB 스키마와 RLS 정책만 준비한다
4. `/myagent` 페이지의 실제 북마크 콘텐츠는 별도 명세로 구현한다
5. 카테고리 필터링은 대시보드의 추천 스킬 목록을 대상으로 한다
6. 기존 스킬 데이터의 카테고리 분류 체계가 새 카테고리(기획/디자인/퍼블리싱/개발/QA)로 전환된다

---

## 6. 제약 조건

1. Constitution에 정의된 DDD 아키텍처 원칙(도메인 → 애플리케이션 → 인프라 계층 분리)을 준수해야 한다
2. `any` 타입 사용을 금지한다
3. TypeScript strict 모드를 준수해야 한다
4. Next.js App Router를 사용한다
5. 카테고리 데이터는 Supabase MCP를 통해 관리한다
6. 기존 사이드바 컴포넌트를 수정하되, 레이아웃 구조(사이드바+헤더)는 유지한다

## 7. 수용 기준 체크리스트

### 사이드바 메뉴
- [ ] "마켓플레이스" 메뉴가 제거되었다
- [ ] "메인" 그룹에 "대시보드"와 "내 에이전트"만 표시된다
- [ ] "내 에이전트" 클릭 시 `/myagent`으로 이동한다
- [ ] 현재 페이지에 해당하는 메뉴가 활성 상태로 표시된다

### 카테고리
- [ ] "카테고리" 그룹에 기획, 디자인, 퍼블리싱, 개발, QA 5개 항목이 표시된다
- [ ] 카테고리 항목은 DB 테이블에서 조회된다
- [ ] 카테고리 클릭 시 대시보드에서 해당 카테고리로 스킬이 필터링된다
- [ ] 필터링 상태가 URL 쿼리 파라미터에 반영된다
- [ ] 활성 카테고리가 사이드바에서 시각적으로 구분된다

### 카테고리 DB
- [ ] `categories` 테이블이 생성되었다 (id, name, slug, icon, sort_order, created_at, updated_at)
- [ ] 시드 데이터로 5개 카테고리가 삽입되었다
- [ ] RLS 정책이 적용되었다 (읽기: 인증 사용자, 쓰기: 관리자)

### 내 에이전트 페이지
- [ ] `/myagent` 경로에 페이지가 존재한다
- [ ] 사이드바+헤더 레이아웃이 루트 페이지와 동일하다
- [ ] 빈 상태 메시지가 표시된다

### 공통
- [ ] 다크 모드에서 정상 동작한다
- [ ] 기존 테스트가 깨지지 않는다 (사이드바 테스트 업데이트 포함)
