# Feature Spec: 로그인/로그아웃

> **Feature ID**: 002
> **Status**: Draft
> **Created**: 2026-03-02
> **Dependencies**: 001-root-page (랜딩 페이지의 로그인/회원가입 버튼)

---

## 1. 기능 개요

이메일/비밀번호 기반의 회원가입, 로그인, 로그아웃 기능을 제공한다.
`@eluocnc.com` 도메인 이메일만 허용하며, 회원가입 시 이메일로 6자리 인증 코드(OTP)를 발송하여 본인 확인을 수행한다.
Supabase Auth를 활용하며, 별도의 `/login`과 `/signup` 페이지를 통해 인증 플로우를 처리한다.
로그아웃 시에는 확인 다이얼로그를 거쳐 세션을 종료한다.

---

## 2. 유저 스토리

### US-001: 신규 사용자의 회원가입
> As a 미가입 사용자,
> I want to 이메일과 비밀번호로 계정을 생성할 수 있기를,
> So that 서비스에 로그인하여 AI 스킬을 활용할 수 있다.

### US-002: 기존 사용자의 로그인
> As a 가입된 사용자,
> I want to 이메일과 비밀번호로 로그인할 수 있기를,
> So that 내 대시보드와 개인화된 기능에 접근할 수 있다.

### US-003: 로그인 사용자의 로그아웃
> As a 로그인된 사용자,
> I want to 안전하게 로그아웃할 수 있기를,
> So that 공유 기기에서 내 계정을 보호할 수 있다.

### US-004: 인증 상태에 따른 접근 제어
> As a 비로그인 사용자,
> I want to 보호된 페이지 접근 시 로그인 페이지로 안내받기를,
> So that 인증 후 원래 접근하려던 페이지로 돌아갈 수 있다.

## 5. 가정 사항

1. Supabase Auth가 프로젝트에 이미 설정되어 있다 (001-root-page에서 `@supabase/ssr` 설치 완료)
2. Supabase Auth의 이메일/비밀번호 인증이 Supabase 대시보드에서 활성화되어 있다
3. Supabase Auth의 이메일 OTP 인증이 활성화되어 있다. 회원가입 시 6자리 인증 코드가 이메일로 발송되며, 코드 입력 완료 후 로그인이 가능하다 (C-12)
4. 비밀번호 찾기/재설정은 별도 명세로 다룬다
5. 소셜 로그인(Google, GitHub 등)은 이 명세 범위에 포함되지 않는다
6. 001-root-page의 랜딩 페이지에서 `/login`, `/signup`으로의 링크가 이미 존재한다

## 6. 제약 조건

1. Constitution에 정의된 DDD 아키텍처 원칙을 준수한다
2. `any` 타입 사용을 금지한다
3. 외부 UI 라이브러리 없이 Tailwind CSS로 구현한다
4. Next.js App Router의 Server Actions 또는 Route Handlers를 활용한다
5. 시크릿/API 키를 코드에 하드코딩하지 않는다 (환경변수 사용)
6. 로그인 실패 시 "이메일 또는 비밀번호가 올바르지 않습니다"와 같은 일반 메시지를 사용한다 (계정 존재 여부 미노출)

## 7. 수용 기준 체크리스트

### 회원가입
- [ ] `/signup` 접속 시 회원가입 폼이 표시된다
- [ ] 이메일, 비밀번호, 비밀번호 확인 필드가 존재한다
- [ ] `@eluocnc.com` 외 도메인 이메일 입력 시 인라인 오류가 표시된다 (C-11)
- [ ] 비밀번호 최소 요건(8자, 영문+숫자) 미충족 시 오류가 표시된다
- [ ] 비밀번호 불일치 시 오류가 표시된다
- [ ] 회원가입 시도 후 `/signup/verify-code` 인증 코드 입력 페이지로 리다이렉트된다 (C-12)
- [ ] 이메일 존재 여부와 관계없이 동일한 안내가 표시된다 (C-06)
- [ ] 제출 중 로딩 상태가 표시되고 중복 클릭이 방지된다
- [ ] "로그인" 링크 클릭 시 `/login`으로 이동한다

### 인증 코드 확인
- [ ] `/signup/verify-code` 접속 시 6자리 코드 입력 필드가 표시된다
- [ ] 가입 시 입력한 이메일 주소가 화면에 표시된다
- [ ] 올바른 코드 입력 시 대시보드로 리다이렉트된다
- [ ] 잘못된 코드 입력 시 오류 메시지가 표시된다
- [ ] "인증 코드 재발송" 클릭 시 코드가 재발송된다
- [ ] 제출 중 로딩 상태가 표시되고 중복 클릭이 방지된다

### 로그인
- [ ] `/login` 접속 시 로그인 폼이 표시된다
- [ ] 이메일, 비밀번호 필드가 존재한다
- [ ] 올바른 자격 증명으로 로그인 시 대시보드로 리다이렉트된다
- [ ] `redirectTo` 파라미터가 있으면 해당 경로로 리다이렉트된다
- [ ] 잘못된 자격 증명 시 일반 오류 메시지가 표시된다
- [ ] 제출 중 로딩 상태가 표시되고 중복 클릭이 방지된다
- [ ] "회원가입" 링크 클릭 시 `/signup`으로 이동한다

### 로그아웃
- [ ] 프로필 아바타 클릭 시 드롭다운 메뉴가 표시된다
- [ ] 드롭다운에 사용자 이메일이 표시된다
- [ ] "로그아웃" 클릭 시 확인 다이얼로그가 표시된다
- [ ] 확인 다이얼로그에서 "취소" 시 아무 동작 없음
- [ ] 확인 다이얼로그에서 "로그아웃" 시 세션 종료 후 랜딩 페이지로 이동
- [ ] 드롭다운 외부 클릭 시 메뉴가 닫힌다

### 접근 제어
- [ ] 비로그인 사용자가 `/` 접근 시 리다이렉트 없이 랜딩 페이지가 표시된다
- [ ] 로그인 사용자가 `/` 접근 시 대시보드가 표시된다
- [ ] 비로그인 사용자가 보호 페이지(화이트리스트 외) 접근 시 `/login`으로 리다이렉트된다
- [ ] 리다이렉트 시 원래 경로가 `redirectTo` 파라미터로 전달된다
- [ ] 로그인 사용자가 `/login`, `/signup` 접근 시 대시보드로 리다이렉트된다

### 공통
- [ ] 다크 모드 정상 동작
- [ ] 반응형 레이아웃 (모바일~데스크탑)
- [ ] 입력 검증이 onBlur + onSubmit 이중 방식으로 동작한다 (C-10)
- [ ] Supabase 레이트 리미팅(429) 시 안내 메시지가 표시된다 (C-08)

## Clarifications

### Round 1 (2026-03-02)

**C-01. 이메일 인증 플로우 (Q1)** ← C-12로 대체됨
> ~~Supabase 이메일 확인 링크 방식~~ → C-12에서 OTP 인증 코드 방식으로 변경.

**C-02. redirectTo 보안 검증 (Q2)**
> `redirectTo` 파라미터는 `/`로 시작하는 같은 도메인의 상대 경로만 허용한다. 절대 URL(`http://`, `https://`, `//` 등)이나 외부 도메인이 포함된 값은 무시하고 기본값 `/`로 리다이렉트한다. 오픈 리다이렉트 공격을 방지한다.

**C-03. DDD 아키텍처 구조 (Q3)**
> Auth 바운디드 컨텍스트의 도메인 계층은 최소화한다. `Email`, `Password` 등 값 객체(Value Object)를 도메인에 정의하여 입력 검증 규칙을 캡슐화하되, 인증 로직 자체는 Supabase Auth에 위임한다. 유스케이스(SignUpUseCase, SignInUseCase, SignOutUseCase)는 application 계층에, Supabase Auth 호출은 infrastructure 계층에 배치한다.
> - `src/auth/domain/value-objects/` : Email, Password
> - `src/auth/application/` : SignUpUseCase, SignInUseCase, SignOutUseCase
> - `src/auth/infrastructure/` : SupabaseAuthRepository

**C-04. 보호된 페이지 범위 (Q4)**
> 공개 경로 화이트리스트 방식을 사용한다. `/`, `/login`, `/signup`, `/signup/verify-code`를 공개 경로로 정의하고, 이 목록에 포함되지 않는 모든 경로는 보호된 페이지로 간주한다. middleware.ts에서 화이트리스트 기반 접근 제어를 구현한다.
> - `/`는 인증 상태에 따라 다른 콘텐츠를 표시하는 특수 경로: 비로그인 시 랜딩 페이지, 로그인 시 대시보드 (001-root-page에서 구현 완료). 리다이렉트가 아닌 조건부 렌더링 방식이다.

**C-05. Header 컴포넌트 변경 방식 (Q5)**
> 기존 Header 컴포넌트는 Server Component로 유지한다. 프로필 아바타와 드롭다운 메뉴 부분만 별도 Client Component(`ProfileDropdown`)로 분리하여, Header에 합성(Composition) 방식으로 포함한다. 로그아웃 확인 다이얼로그도 ProfileDropdown 내부에서 관리한다.

### Round 2 (2026-03-02)

**C-06. 회원가입 시 이메일 존재 여부 노출 방지 (Q6)**
> 보안 우선 원칙을 적용한다. 이미 가입된 이메일로 회원가입을 시도하더라도, 실제 가입 여부와 관계없이 동일한 응답("이메일 확인 메일을 발송했습니다")을 표시하여 계정 존재 여부를 노출하지 않는다.
> - FR-08 변경: "이미 가입된 이메일" 오류 메시지 대신, 모든 회원가입 시도에 동일한 확인 안내를 표시한다.
> - NFR-02와의 일관성 확보: 회원가입/로그인 모두에서 계정 존재 여부를 추론할 수 없도록 한다.

**C-07. 비밀번호 규칙 상세 (Q7)**
> 최소 규칙만 적용한다:
> - 최소 8자, 최대 72자 (bcrypt 호환)
> - 영문(대소문자 무관) 1자 이상 포함
> - 숫자 1자 이상 포함
> - 특수문자는 허용하되 필수가 아님
> - FR-31, FR-32 유지, FR-31에 최대 72자 조건 추가

**C-08. 브루트포스 방어 (Q8)**
> Supabase Auth의 기본 레이트 리미팅에 위임한다. 클라이언트 측에서는 중복 제출 방지(FR-11, FR-20)만 처리하고, 서버 측 레이트 리미팅은 Supabase 기본 설정을 신뢰한다. Supabase에서 429 응답이 오면 "잠시 후 다시 시도해주세요" 메시지를 표시한다.

**C-09. 세션 만료 시 동작 (Q9)**
> 조용한 리다이렉트 방식을 사용한다. 세션이 만료된 상태에서 보호된 페이지에 접근하면, 별도의 알림 없이 `/login?redirectTo={현재경로}`로 리다이렉트한다. 이는 FR-28의 기존 접근 제어 플로우와 동일하게 처리한다.

**C-10. 입력 검증 타이밍 (Q10)**
> onBlur + onSubmit 이중 검증 방식을 사용한다:
> - onBlur: 사용자가 입력 필드를 떠날 때 해당 필드만 검증하여 인라인 오류를 표시한다.
> - onSubmit: 폼 제출 시 모든 필드를 재검증하고, 첫 번째 오류 필드로 포커스를 이동한다.
> - 입력 중(onChange)에는 이미 표시된 오류가 있을 때만 실시간으로 해제한다.
